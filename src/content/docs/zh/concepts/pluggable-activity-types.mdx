# 可插拔活动类型

流程是一个节点图。大多数节点在进程内运行一个 WASM 组件，但某些
节点会将工作交给一个**独立的运行时**——SoRLa、OperaLa，以及（在
未来的）其他运行时——并等待结果。这些就是**活动类型**：
原生流程节点，其职责是通过消息总线分派到外部运行时。

如今平台提供以下分派活动类型：

| 节点 | 分派到 | 状态 |
| --- | --- | --- |
| `sorla.call` | SoRLa 运行时 | 已上线 |
| `operala.call` | OperaLa 运行时 | 已上线 |
| `telco-x.call` | 一个 telco-x 运行时 | 已接线，在运行时存在前处于惰性状态 |

## 分派如何工作

分派节点本身不运行业务逻辑。它打包节点的
输入，通过 NATS 向运行时发布一个请求，然后要么：

- **等待（awaits）**响应（流程暂停，并在结果
  到达时恢复），要么
- **发后即忘（fires and forgets）**（流程立即继续）。

所有分派节点共享一个机制——相同的请求/响应契约和
相同的关联（correlation）处理——因此无论它们针对哪个
运行时，行为都一致。

<Aside type="note">
分派节点在其目标运行时实际部署并
可在总线上访问之前是*惰性的*。例如，`telco-x.call`
已完全接入 runner 和 Designer 两者，但在 telco-x 运行时
运行之前它什么也不做——这正是“wire-ready（接线就绪）”的含义。
</Aside>

## 这对 Designer 为何重要

Designer 的流程构建器将其调色板锚定在一个**能力目录（capability catalog）**上。
每个活动类型在该目录中作为一个可分派能力出现，这
意味着当用户描述属于这些运行时之一的工作时，AI 流程构建器可以放置一个
`sorla.call` 或 `operala.call` 节点——就像它
放置任何其他节点一样。

## 添加一个新的活动类型

设计目标是让添加一个全新的活动类型几乎不需要任何成本：

1. 在 Designer 的基线能力目录中**一条目录条目**，标记
   为分派能力。这使该节点能被 AI 流程
   构建器发现。
2. 在 runner 中**一行代码**，将该节点识别为原生分派
   操作。

runner 已经知道如何通用地执行任何分派节点，因此无需
新的执行代码——该节点只是针对一个不同的运行时
名称。当该运行时之后被部署时，先前惰性的节点便会变为
上线状态，无需任何进一步的更改。

<Aside type="tip">
这与 `sorla.call` 和 `operala.call` 所使用的模式相同，也是
`telco-x.call` 在其运行时之前被添加的方式。之后添加运行时会
“点亮”该节点，而无需再次触碰流程编辑器或 runner。
</Aside>
