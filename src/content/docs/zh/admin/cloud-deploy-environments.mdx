# 云部署环境

**部署环境**是平台操作员为租户配置的具名云目的地——Greentic Cloud、AWS、
Azure 或 GCP。一旦环境存在，租户管理员便可直接从 Designer 将已发布的 pack
部署到该环境，而无需接触云凭据或
deployer CLI。

本页涵盖使其生效的三个部分：

1. **环境（Environments）**——操作员为每个租户（以及可选的每个团队）配置的具名目的地。
2. **部署插件（Deploy plugins）**——租户被允许使用哪些部署目标。
3. **从 Designer 进行云部署**——租户管理员的自助流程。

## 角色一览

| 操作 | 由谁执行 |
| --- | --- |
| 创建 / 编辑 / 删除环境 | 仅平台操作员 |
| 配置自带云（bring-your-own-cloud）凭据 | 仅平台操作员 |
| 为租户启用 / 禁用部署目标 | 仅平台操作员 |
| 列出环境并将 pack 部署到某一环境 | 租户管理员（在 Designer 中） |

这种划分是刻意为之的：操作员拥有云账户和凭据，
租户拥有决定*交付什么*以及*何时交付*的权力。

## 环境

在管理控制台中，打开一个租户并进入 **Environments** 选项卡。每个
环境具有：

- 一个**名称**（在租户内唯一），
- 一个**目标（target）**——`greentic_cloud`、`aws`、`azure` 或 `gcp`，
- 一个可选的**区域（region）**，
- 一个可选的**作用域（scope）**——租户范围（对每个团队可见）或固定到单个团队，
- 一个**状态（status）**——`idle`、`provisioning`、`live` 或 `error`。

### Greentic Cloud 与自带云

- **Greentic Cloud**（`greentic_cloud`）使用平台自身的托管角色进行部署。
  无需任何每租户凭据——环境一经创建即可使用。
- **AWS / Azure / GCP** 属于*自带云*（BYOC）。在租户能够
  部署到其中之一之前，操作员必须为该环境附加云凭据。
  在此之前，部署尝试会被拒绝，并附带清晰的“请先配置凭据”
  提示。

<Aside type="note">
BYOC 凭据作为环境上的密钥引用存储，
绝不会返回给 Designer 或展示给租户用户。Designer 只会
知道凭据是否存在，而不会知道其值。
</Aside>

### 每团队环境

环境可以限定到单个团队，而不是整个租户。
团队范围的环境仅对该团队的成员可见，且其
BYOC 凭据被隔离到所属团队的密钥作用域中。将
作用域保留为*租户范围*，即可在每个团队之间共享同一个环境。

## 部署插件

并非每个租户都应能够交付到每个云。**部署插件**允许
操作员按租户和按团队逐个开启或关闭各个部署目标。

在 **Environments** 选项卡中，部署目标开关控制创建环境时
会出现哪些目标。可用目标的目录
来源于 deployer 本身（只有实际能够运行的目标才会被
提供——`greentic_cloud`、`aws`、`azure`、`gcp`），因此该列表始终与
平台能够部署的内容相匹配。

解析顺序为 **团队 → 租户 → 默认启用**：团队级别的设置优先于
租户级别的设置，任何未设置的项默认启用。
为已禁用的目标创建环境会被阻止。

## 从 Designer 进行云部署

一旦操作员配置了至少一个环境，租户
管理员便可从 Designer 进行部署：

1. 打开 **Settings → Credentials**。
2. 点击 **Cloud deploy**。
3. 输入要交付的 **pack 引用**（例如 `my-pack@1.0.0`）。
4. 选择一个目标环境并点击 **Deploy**。

部署被加入队列，环境进入 `provisioning` 状态。如果
所选环境是一个尚未配置凭据的 BYOC 目标，
该部署会被拒绝并附带面向操作员的原因，而不是
默默失败。

<Aside type="tip">
**Cloud deploy** 按钮仅对租户管理员显示。普通
成员不会看到它，且底层 API 会拒绝他们的调用。
</Aside>

### 独立 Designer

当 Designer 在没有管理服务器的情况下运行（独立模式）时，云部署
根本不会被提供：环境列表为空，面板会说明
未配置任何管理服务器。通过部署
向导进行的本地部署和基于 bundle 的部署不受影响。
