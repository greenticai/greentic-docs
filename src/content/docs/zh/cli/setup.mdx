## 概览

`gtc setup` 命令用于配置你的 Greentic bundle 中的 provider。它会为每个已启用的 provider 运行 setup flow，收集凭据并配置 webhook。

## 用法

```bash
gtc setup [OPTIONS] <BUNDLE_PATH>
```

## 选项

| 选项 | 说明 |
|--------|-------------|
| `--answers <FILE>` | answers 文件路径（非交互模式） |
| `--dry-run` | 预览 setup 过程而不进行实际更改 |
| `--emit-answers <FILE>` | 生成 answers 模板 |
| `--advanced` | 在 answers 模板中包含可选字段 |
| `--provider <NAME>` | 仅设置特定 provider |
| `--skip <NAME>` | 跳过特定 provider |
| `-v, --verbose` | 启用详细输出 |

## 交互式 Setup

以交互方式运行 setup，CLI 会针对每个 provider 提示你输入：

```bash
gtc setup ./my-bundle
```

CLI 会提示你提供：
- Provider 凭据（API keys、tokens）
- 用于 webhook 的公开 URL
- 可选配置项

## 非交互式 Setup

### Demo Catalog Flows

[`greentic-demo`](https://github.com/greenticai/greentic-demo) catalog 中的每个 demo，都会在其 pack 旁边发布一个 `*-setup-answers.json` 文档。你可以将它应用到使用 `gtc wizard` 创建的 bundle：

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

同样的模式适用于任何 demo，只需替换 demo ID 即可。完整 catalog、说明以及所需凭据请参见[运行 Demo](/zh/getting-started/running-demos/)。

你也可以让 `gtc setup` 直接处理已发布的 `.gtbundle` artifact：

```bash
gtc setup https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo.gtbundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo-setup-answers.json
```

<Aside type="note">
`gtc setup` 会从 `tenants/` 目录自动检测 bundle 的 tenant，因此即使 bundle 不是用默认的 `demo` tenant 创建的，secrets 也会写入正确的 tenant URI。它还会对 bundle 中的每个 pack 运行**依赖 capability 校验**，如果仍有任何必需 capability（例如 `greentic:state/state-store`）未满足，就会发出警告。
</Aside>

### 生成 Answers 模板

首先，生成包含所有必填字段的模板：

```bash
# Basic template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Full template (including optional fields)
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle
```

## Provider 专用配置

### Telegram

```json
{
  "messaging-telegram": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
  }
}
```

**如何获取凭据：**
1. 在 Telegram 中联系 [@BotFather](https://t.me/botfather)
2. 发送 `/newbot` 并按照提示操作
3. 复制 bot token

### Slack

```json
{
  "messaging-slack": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "xoxb-xxx-xxx-xxx",
    "slack_app_id": "A07XXXXXX",
    "slack_configuration_token": "xoxe.xoxp-xxx"
  }
}
```

**如何获取凭据：**
1. 访问 [api.slack.com/apps](https://api.slack.com/apps)
2. 创建一个新应用或选择现有应用
3. 在 OAuth & Permissions 中获取 Bot Token
4. 在 Basic Information 中获取 App ID
5. 在 App Manifest 中获取 Configuration Token

## 公开 URL 配置

<Aside type="caution">
基于 webhook 的 provider 需要一个可公开访问的 URL。使用 `gtc start` 配合 ngrok 或 cloudflared 时，`public_base_url` 会被自动替换。
</Aside>

### 使用 ngrok

```bash
# Start ngrok in separate terminal
ngrok http 8080

# Copy the HTTPS URL (e.g., https://abc123.ngrok-free.app)
# Use this as public_base_url in answers
```

### 使用 Cloudflared

```bash
# Cloudflared generates URL automatically with gtc start
gtc start ./my-bundle --cloudflared on
```

## Bundle Capabilities

`bundle.yaml` 文件可以声明 capabilities，以在 setup 期间启用额外行为。Capabilities 以 capability 标识符数组的形式列出。

### Bundle Assets (`greentic.cap.bundle_assets.read.v1`)

当 `bundle.yaml` 中存在 `greentic.cap.bundle_assets.read.v1` capability 时，`gtc setup` 会在 bundle 根目录创建一个 `assets/` 目录，并将 provider packs 中的 asset 文件 scaffold 到该目录。

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

具体来说，provider packs 中的 assets（例如 `assets/webchat-gui/` 条目）会被复制到 bundle 级别的 `assets/` 目录中。这让你可以在不修改原始 pack 内容的情况下，自定义 provider assets，例如 webchat-gui 皮肤和配置文件。

**工作方式：**

- 在 setup 期间，包含 asset 目录的 provider packs（例如 `assets/webchat-gui/`）会将这些文件 scaffold 到 bundle 根目录下的 `assets/` 目录。
- 之后你可以自由编辑这些文件。运行时会优先使用 bundle 级别的 asset 文件，而不是 pack 默认值。
- 这对于自定义 webchat widget 的外观和体验、调整配置或应用品牌皮肤非常有用。

<Aside type="tip">
setup 完成 asset scaffold 后，你可以安全地修改它们。运行时加载 assets 时，你在 bundle 级别所做的自定义会优先于 pack 默认值。
</Aside>

## Setup Flow 执行过程

当你运行 `gtc setup` 时，它会：

<Steps>

1. **加载 bundle 配置**

   读取 `greentic.demo.yaml` 以查找 providers。

2. **校验 answers**

   检查所有必填字段是否存在。

3. **运行 setup flows**

   执行每个 provider 的 `setup_flow`（例如 webhook 注册）。

4. **存储凭据**

   将凭据保存到已配置的 secrets store。

5. **执行验证**

   运行 `verify_flow` 以确认 setup 是否成功。

</Steps>

## 选择性 Setup

### 设置单个 Provider

```bash
gtc setup --provider messaging-telegram ./my-bundle
```

### 跳过特定 Provider

```bash
gtc setup --skip messaging-teams ./my-bundle
```

### 重新运行 Setup

```bash
# Force re-setup (useful after URL changes)
gtc start ./my-bundle --force-setup
```

## 故障排查

### Webhook 注册失败

```
Error: Failed to register webhook: 401 Unauthorized
```

检查你的 bot token 是否正确且未过期。

### 缺少必填字段

```
Error: Missing required field 'bot_token' for messaging-telegram
```

确认你的 answers 文件中包含所有必填字段。

### 无法访问公开 URL

```
Error: Webhook verification failed: Connection refused
```

确认你的公开 URL 可从互联网访问，并检查 ngrok/cloudflared 是否正在运行。

### 找不到凭据

```
Error: Secret 'slack_bot_token' not found
```

运行 `gtc setup` 来配置该 provider，或检查你的 answers 文件。

## 最佳实践

1. **在 CI/CD 中使用 answers 文件** - 避免在自动化流水线中使用交互式提示
2. **妥善保护凭据** - 不要将包含真实 token 的 answers 文件提交到 git
3. **使用环境变量** - 在 answers 中通过 `$ENV_VAR` 引用 secrets
4. **先使用 dry-run 测试** - 在实际更改前先预览 setup 结果
5. **URL 变化后重新生成** - 当你的公开 URL 发生变化时重新运行 setup
