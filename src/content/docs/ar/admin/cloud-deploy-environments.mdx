# بيئات النشر السحابي

**بيئة النشر** هي وجهة سحابية مُسماة — Greentic Cloud أو AWS أو
Azure أو GCP — يقوم مشغل المنصة بتكوينها لمستأجر. وبمجرد وجود
بيئة، يمكن لمسؤول المستأجر نشر حزمة منشورة إليها
مباشرة من خلال Designer، دون الحاجة إلى التعامل مع بيانات اعتماد السحابة أو
واجهة سطر أوامر أداة النشر.

تغطي هذه الصفحة الأجزاء الثلاثة التي تجعل ذلك ممكنًا:

1. **البيئات** — الوجهات المُسماة التي يكوّنها المشغل لكل مستأجر (وبشكل اختياري لكل فريق).
2. **إضافات النشر** — أهداف النشر التي يُسمح للمستأجر باستخدامها.
3. **النشر السحابي من Designer** — تدفق الخدمة الذاتية لمسؤول المستأجر.

## الأدوار بإيجاز

| الإجراء | من |
| --- | --- |
| إنشاء / تعديل / حذف بيئة | مشغل المنصة فقط |
| تكوين بيانات اعتماد السحابة الخاصة بك | مشغل المنصة فقط |
| تمكين / تعطيل أهداف النشر لمستأجر | مشغل المنصة فقط |
| سرد البيئات ونشر حزمة إلى إحداها | مسؤول المستأجر (في Designer) |

هذا التقسيم متعمّد: المشغلون يملكون حسابات السحابة وبيانات الاعتماد،
والمستأجرون يملكون قرار *ماذا* يُشحن و*متى*.

## البيئات

في وحدة تحكم الإدارة، افتح مستأجرًا وانتقل إلى علامة التبويب **Environments**. لكل
بيئة:

- **اسم** (فريد ضمن المستأجر)،
- **هدف** — `greentic_cloud` أو `aws` أو `azure` أو `gcp`،
- **منطقة** اختيارية،
- **نطاق** اختياري — على مستوى المستأجر بأكمله (مرئي لكل فريق) أو مثبّت على فريق واحد،
- **حالة** — `idle` أو `provisioning` أو `live` أو `error`.

### Greentic Cloud مقابل السحابة الخاصة بك

- **Greentic Cloud** (`greentic_cloud`) ينشر باستخدام دور
  المنصة المُدار الخاص بها. لا حاجة إلى بيانات اعتماد لكل مستأجر — فهو
  يعمل بمجرد إنشاء البيئة.
- **AWS / Azure / GCP** هي *سحابة خاصة بك* (BYOC). قبل أن يتمكن المستأجر من
  النشر إلى إحداها، يجب على المشغل إرفاق بيانات اعتماد السحابة لتلك البيئة.
  وحتى ذلك الحين، تُرفض أي محاولة نشر برسالة واضحة تقول "قم بتكوين بيانات الاعتماد
  أولاً".

<Aside type="note">
تُخزَّن بيانات اعتماد BYOC كمرجع سري على البيئة ولا تُعاد
أبدًا إلى Designer أو تُعرض على مستخدمي المستأجر. لا يعرف Designer
سوى ما إذا كانت بيانات الاعتماد موجودة، وليس قيمتها.
</Aside>

### البيئات لكل فريق

يمكن تحديد نطاق البيئة لفريق واحد بدلاً من المستأجر بأكمله.
البيئات المحددة بنطاق فريق مرئية فقط لأعضاء ذلك الفريق، وتكون
بيانات اعتماد BYOC الخاصة بها معزولة في النطاق السري للفريق المالك. اترك
النطاق *على مستوى المستأجر* لمشاركة بيئة واحدة عبر كل فريق.

## إضافات النشر

ليس كل مستأجر ينبغي أن يكون قادرًا على الشحن إلى كل سحابة. **إضافات النشر** تتيح
للمشغل تشغيل أو إيقاف أهداف نشر فردية، لكل مستأجر ولكل فريق.

في علامة التبويب **Environments**، تتحكم مفاتيح تبديل أهداف النشر في الأهداف التي
تظهر عند إنشاء بيئة. ويُستمد كتالوج الأهداف المتاحة من
أداة النشر نفسها (تُعرض فقط الأهداف التي يمكن تشغيلها فعليًا —
`greentic_cloud` و`aws` و`azure` و`gcp`)، بحيث تتطابق القائمة دائمًا
مع ما يمكن للمنصة نشره.

الحل هو **الفريق ← المستأجر ← مُمكَّن افتراضيًا**: يفوز إعداد على مستوى الفريق
على الإعداد على مستوى المستأجر، وأي شيء يُترك دون تعيين يكون مُمكَّنًا افتراضيًا.
يُحظر إنشاء بيئة لهدف معطّل.

## النشر السحابي من Designer

بمجرد أن يكوّن المشغل بيئة واحدة على الأقل، يمكن لمسؤول
المستأجر النشر من Designer:

1. افتح **Settings → Credentials**.
2. انقر فوق **Cloud deploy**.
3. أدخل **مرجع الحزمة** المراد شحنها (على سبيل المثال `my-pack@1.0.0`).
4. اختر بيئة هدف وانقر فوق **Deploy**.

يُدرَج النشر في قائمة الانتظار وتنتقل البيئة إلى `provisioning`. وإذا كانت
البيئة المختارة هدف BYOC لم تُكوَّن بيانات اعتماده
بعد، يُرفض النشر مع السبب الموجّه إلى المشغل بدلاً من
الفشل بصمت.

<Aside type="tip">
يظهر زر **Cloud deploy** فقط لمسؤولي المستأجرين. الأعضاء
العاديون لا يرونه، وواجهة برمجة التطبيقات الأساسية ترفض استدعاءاتهم.
</Aside>

### Designer المستقل

عندما يعمل Designer دون خادم إدارة (الوضع المستقل)، لا يُعرض النشر
السحابي ببساطة: تكون قائمة البيئات فارغة وتوضح اللوحة
أنه لم يُكوَّن أي خادم إدارة. ولا تتأثر عمليات النشر المحلية والمستندة إلى الحزم عبر
معالج النشر.
