## Resumen

El comando `gtc wizard` crea nuevos bundles de Greentic a partir de respuestas del wizard. Genera la estructura completa del bundle, incluidos archivos de configuración, configuraciones de proveedores y plantillas de aplicaciones.

## Uso

```bash
gtc wizard [OPTIONS]
```

## Opciones

| Opción | Descripción |
|--------|-------------|
| `--answers <FILE \| URL>` | Ruta o URL al archivo de respuestas del wizard (JSON o YAML). Admite rutas locales y URLs remotas `http://` / `https://`. |
| `--dry-run` | Previsualiza los archivos generados sin escribirlos |
| `--output <DIR>` | Directorio de salida (predeterminado: directorio actual) |
| `--template <NAME>` | Usa una plantilla específica |
| `-v, --verbose` | Habilita salida detallada |

## Archivo de Respuestas del Wizard

### Formato JSON

```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0",
    "description": "Customer support digital worker"
  },
  "providers": {
    "messaging": ["telegram", "slack"],
    "events": ["webhook", "timer"]
  },
  "apps": [
    {
      "name": "support-bot",
      "template": "customer-service"
    }
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [
        {
          "id": "default",
          "channels": ["telegram", "slack"]
        }
      ]
    }
  ]
}
```

### Formato YAML

```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"
  description: Customer support digital worker

providers:
  messaging:
    - telegram
    - slack
  events:
    - webhook
    - timer

apps:
  - name: support-bot
    template: customer-service

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - telegram
          - slack
```

### Capacidades

El array `capabilities` en el archivo de respuestas del wizard controla las funciones a nivel de bundle que se habilitan cuando se crea el bundle. Las capacidades determinan qué generación adicional realiza el wizard más allá de la estructura estándar del bundle.

```json title="create-answers.json"
{
  "capabilities": ["greentic.cap.bundle_assets.read.v1"]
}
```

**Capacidades disponibles:**

| Capacidad | Descripción |
|------------|-------------|
| `greentic.cap.bundle_assets.read.v1` | Habilita reemplazos de assets a nivel de bundle. Cuando está presente, el wizard genera automáticamente los assets de skin de webchat-gui desde los provider packs en el directorio `assets/` del bundle. |

Cuando se incluye `greentic.cap.bundle_assets.read.v1`, el bundle generado contendrá un directorio `assets/` con archivos copiados desde provider packs (por ejemplo, skins y configuración de webchat-gui). Estos archivos a nivel de bundle pueden personalizarse y reemplazarán los valores predeterminados del pack en tiempo de ejecución.

<Aside type="tip">
El array `capabilities` se escribe en `bundle.yaml` y `gtc setup` también lo respeta al configurar el bundle más tarde. Consulta la [sección de capacidades de gtc setup](/es/cli/setup/#bundle-capabilities) para conocer los detalles del comportamiento en tiempo de ejecución.
</Aside>

Al ejecutar el wizard se genera:

```
my-digital-worker/

├── providers/
│   ├── messaging/

│   └── events/

│       └── teams/

```

## Ejemplos

### Bundles del Catálogo de Demos

El repositorio [`greentic-demo`](https://github.com/greenticai/greentic-demo) publica un `*-create-answers.json` y un `*-setup-answers.json` para cada demo en sus artefactos de release. Úsalos directamente para crear un bundle local que puedas seguir editando después:

```bash
gtc wizard --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-create-answers.json
```

Luego continúa con la configuración y el arranque:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
gtc start ./quickstart-demo-bundle
```

Sustituye el ID de la demo por cualquiera de las demos publicadas (`quickstart`, `quickstart-event`, `hr-onboarding`, `helpdesk-itsm`, `sales-crm`, `supply-chain`, `incident`, `redbutton`, `cloud-deploy-demo`, `weather-mcp-demo`, ...). El catálogo completo con descripciones y credenciales requeridas está en la página [Ejecutar Demos](/es/getting-started/running-demos/).

<Aside type="tip">
El wizard realiza **resolución de dependencias basada en capacidades**. Cuando el documento create-answers de la demo hace referencia a un proveedor que declara `required_capabilities` (por ejemplo, proveedores de mensajería que necesitan `greentic:state/state-store`), el wizard añade automáticamente un pack que lo satisface, como `state-memory`, al bundle generado; no tienes que listarlo manualmente.
</Aside>

### Uso Básico

```bash
# Interactive mode
gtc wizard

# With answers file
gtc wizard --answers wizard-answers.yaml
```

### Modo de Previsualización

```bash
# See what would be generated
gtc wizard --answers wizard-answers.yaml --dry-run
```

### Archivo de Respuestas Remoto

La bandera `--answers` acepta URLs `http://` y `https://`, lo que facilita
almacenar documentos de respuestas compartidos en una ubicación central y referenciarlos desde pipelines de CI/CD.

```bash
# Fetch answers from a remote URL
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml

# Combine with dry-run to preview before generating
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml --dry-run
```

<Aside type="tip">
El soporte para URLs remotas es útil en pipelines de CI/CD donde las respuestas del wizard se almacenan en un repositorio compartido o en un servidor de artefactos y se descargan en tiempo de build.
</Aside>

### Directorio de Salida Personalizado

```bash
gtc wizard --answers wizard-answers.yaml --output ./my-project
```

### Uso de Plantillas

```bash
# List available templates
gtc wizard --list-templates

# Use specific template
gtc wizard --template customer-service --answers answers.yaml
```

## Plantillas Disponibles

| Plantilla | Descripción |
|----------|-------------|
| `minimal` | Bundle básico con un proveedor |
| `customer-service` | Bot de FAQ con escalado |
| `helpdesk` | Soporte de IT con integración de tickets |
| `multi-channel` | Múltiples canales de mensajería |

## Capacidades del Bundle

El wizard solicita capacidades opcionales del bundle que habilitan funciones adicionales. Cada capacidad desbloquea un conjunto específico de assets y comportamientos en el bundle generado.

### Prompts Interactivos

Al ejecutar `gtc wizard` de forma interactiva, verás:

```
? Enable bundle-level assets (./assets/)? [Y/n]
? Enable OAuth login for WebChat GUI? [y/N]
? Enable multi-language for WebChat GUI? [y/N]
? Enable embeddable WebChat widget? [y/N]
```

### Referencia de Capacidades

| Prompt | ID de Capacidad | Qué Hace |
|--------|-----------------|-----------|
| Enable bundle-level assets | `greentic.cap.bundle_assets.read.v1` | Crea el directorio `./assets/` y genera archivos de skin predeterminados desde provider packs en directorios por tenant |
| Enable OAuth login for WebChat GUI | `greentic.cap.webchat.oauth.v1` | Añade configuración de proveedor de autenticación OAuth/OIDC a los archivos de configuración del tenant |
| Enable multi-language for WebChat GUI | `greentic.cap.webchat.i18n.v1` | Habilita el selector de idioma en WebChat y genera archivos JSON de locale en `./assets/` |
| Enable embeddable WebChat widget | `greentic.cap.webchat.embed.v1` | Genera fragmentos HTML de embed (enlace de página completa, iframe inline, burbuja de chat) por tenant |

### Definir Capacidades en el Archivo de Respuestas

En lugar de responder prompts de forma interactiva, puedes definir capacidades directamente en el archivo de respuestas añadiendo un array `capabilities`:

<Tabs>
  <TabItem label="JSON">
```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0"
  },
  "providers": {
    "messaging": ["webchat", "telegram"]
  },
  "capabilities": [
    "greentic.cap.bundle_assets.read.v1",
    "greentic.cap.webchat.oauth.v1",
    "greentic.cap.webchat.i18n.v1",
    "greentic.cap.webchat.embed.v1"
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [{ "id": "default", "channels": ["webchat"] }]
    }
  ]
}
```
  </TabItem>
  <TabItem label="YAML">
```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"

providers:
  messaging:
    - webchat
    - telegram

capabilities:
  - greentic.cap.bundle_assets.read.v1
  - greentic.cap.webchat.oauth.v1
  - greentic.cap.webchat.i18n.v1
  - greentic.cap.webchat.embed.v1

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - webchat
```
  </TabItem>
</Tabs>

<Aside type="tip">
Las capacidades son aditivas. Habilitar `greentic.cap.webchat.embed.v1` requiere implícitamente `greentic.cap.bundle_assets.read.v1` porque los fragmentos de embed se escriben en el directorio `./assets/`. El wizard gestiona esta dependencia automáticamente.
</Aside>

Para ver todos los detalles sobre capacidades, comportamiento de generación y el mecanismo de overlay, consulta [Bundle Assets](/es/greentic-docs/concepts/bundle-assets/).

## Flujo de Trabajo

<Steps>

1. **Crear archivo de respuestas**

   Define la configuración de tu proyecto en JSON o YAML.

2. **Ejecutar wizard**

   ```bash
   gtc wizard --answers wizard-answers.yaml
   ```

3. **Revisar archivos generados**

   Comprueba la estructura del bundle generado.

4. **Configurar proveedores**

   ```bash
   gtc setup ./my-digital-worker
   ```

5. **Iniciar el runtime**

   ```bash
   gtc start ./my-digital-worker
   ```

</Steps>

## Personalizar los Flujos Generados

El wizard genera plantillas de flujo básicas. Personalízalas después de la generación:

```yaml title="apps/support-bot/flows/on_message.ygtc"
name: on_message
version: "1.0"
description: Handle incoming messages

nodes:
  # Add your custom nodes here
  - id: analyze
    type: llm
    config:
      model: "gpt-4"
      prompt: "Analyze: {{message}}"
    next: respond

  - id: respond
    type: reply
    config:
      message: "{{analysis_result}}"

triggers:
  - type: message
    target: analyze
```

<Aside type="tip">
Usa primero `--dry-run` para previsualizar la estructura generada antes de escribirla en disco.
</Aside>

## Component Scaffolding: `http_client` Capability

When scaffolding a new WASM component (for example via `greentic-component new`), the wizard can mark the component as needing outbound HTTP by setting `http_client: true`. This writes the capability declaration into the component manifest.

<Aside type="caution">
**Desktop runner HTTP gate.** Declaring `http_client: true` is a capability _request_ — the runtime must grant it before the component can actually open outbound sockets. On the **desktop runner profile**, honouring this manifest flag has historically been a no-op: components that declared `http_client: true` had no `wasi:http/outgoing-handler` implementation linked, and runtime HTTP calls failed.

Starting with Greentic runner `N.N.N` (landing with the desktop HTTP gate fix, not yet released), the desktop profile will honour the manifest and grant `wasi:http/outgoing-handler@0.2.0` automatically when a component declares `http_client: true`. Until that release is available, use the workaround below. {/* TODO: confirm runner version once the desktop HTTP gate fix lands */}

**Workaround until the fix ships:** ship your component with a custom WIT world that imports `wasi:http/outgoing-handler@0.2.0` directly, rather than relying on the scaffolded manifest flag. Once you are on a runner that includes the desktop HTTP gate manifest-honouring fix, you can drop the custom world and use `http_client: true` as intended.

For components that emit Adaptive Cards via HTTP fetches (e.g. RAG lookups before returning an `attachments[]` array), see the [WebChat provider's Attachment Envelope Shape](/es/providers/messaging/webchat/#attachment-envelope-shape) section for the expected component-output layout.
</Aside>

## Solución de Problemas

### Archivo de Respuestas Inválido

```
Error: Failed to parse answers file
```

Asegúrate de que tu JSON/YAML sea válido. Usa un linter para comprobar la sintaxis.

### Faltan Campos Requeridos

```
Error: Missing required field: project.name
```

Comprueba que todos los campos requeridos estén presentes en tu archivo de respuestas.

### Plantilla No Encontrada

```
Error: Template 'unknown' not found
```

Usa `gtc wizard --list-templates` para ver las plantillas disponibles.
