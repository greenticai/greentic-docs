## Resumen

El comando `gtc setup` configura los proveedores en tu bundle de Greentic. Ejecuta flujos de configuración para cada proveedor habilitado, recopila credenciales y configura webhooks.

## Uso

```bash
gtc setup [OPTIONS] <BUNDLE_PATH>
```

## Opciones

| Opción | Descripción |
|--------|-------------|
| `--answers <FILE>` | Ruta al archivo de respuestas (no interactivo) |
| `--dry-run` | Previsualiza la configuración sin realizar cambios |
| `--emit-answers <FILE>` | Genera una plantilla de respuestas |
| `--advanced` | Incluye campos opcionales en la plantilla de respuestas |
| `--provider <NAME>` | Configura solo un proveedor específico |
| `--skip <NAME>` | Omite un proveedor específico |
| `-v, --verbose` | Habilita salida detallada |

## Configuración Interactiva

Ejecuta la configuración de forma interactiva para que se te solicite cada proveedor:

```bash
gtc setup ./my-bundle
```

La CLI solicitará:
- Credenciales del proveedor (API keys, tokens)
- URL pública para webhooks
- Opciones de configuración opcionales

## Configuración No Interactiva

### Flujos del Catálogo de Demos

Cada demo del catálogo [`greentic-demo`](https://github.com/greenticai/greentic-demo) publica un documento `*-setup-answers.json` junto con su pack. Aplícalo a un bundle que hayas creado con `gtc wizard`:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

El mismo patrón funciona para cualquier demo; solo sustituye el ID de la demo. Consulta [Ejecutar Demos](/es/getting-started/running-demos/) para ver el catálogo completo con descripciones y credenciales requeridas.

También puedes apuntar `gtc setup` directamente a un artefacto `.gtbundle` publicado:

```bash
gtc setup https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo.gtbundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo-setup-answers.json
```

<Aside type="note">
`gtc setup` detecta automáticamente el tenant del bundle a partir del directorio `tenants/`, por lo que los secretos se escriben en el URI de tenant correcto incluso cuando el bundle no se creó con el tenant predeterminado `demo`. También ejecuta **validación de capacidades de dependencias** en cada pack del bundle y avisa si alguna capacidad requerida (por ejemplo, `greentic:state/state-store`) sigue sin satisfacerse.
</Aside>

### Generar Plantilla de Respuestas

Primero, genera una plantilla con todos los campos requeridos:

```bash
# Basic template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Full template (including optional fields)
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle
```

## Configuración Específica por Proveedor

### Telegram

```json
{
  "messaging-telegram": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
  }
}
```

**Cómo obtener credenciales:**
1. Envía un mensaje a [@BotFather](https://t.me/botfather) en Telegram
2. Envía `/newbot` y sigue las indicaciones
3. Copia el token del bot

### Slack

```json
{
  "messaging-slack": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "xoxb-xxx-xxx-xxx",
    "slack_app_id": "A07XXXXXX",
    "slack_configuration_token": "xoxe.xoxp-xxx"
  }
}
```

**Cómo obtener credenciales:**
1. Ve a [api.slack.com/apps](https://api.slack.com/apps)
2. Crea una app nueva o selecciona una existente
3. Obtén el Bot Token en OAuth & Permissions
4. Obtén el App ID en Basic Information
5. Obtén el Configuration Token en App Manifest

### Microsoft Teams

```json

  "messaging-teams": {

  }
}
```

## Configuración de URL Pública

<Aside type="caution">
Los proveedores basados en webhooks requieren una URL accesible públicamente. `public_base_url` se reemplazará automáticamente al usar ngrok o cloudflared con `gtc start`.
</Aside>

### Usar ngrok

```bash
# Start ngrok in separate terminal
ngrok http 8080

# Copy the HTTPS URL (e.g., https://abc123.ngrok-free.app)
# Use this as public_base_url in answers
```

### Usar Cloudflared

```bash
# Cloudflared generates URL automatically with gtc start
gtc start ./my-bundle --cloudflared on
```

## Capacidades del Bundle

El archivo `bundle.yaml` puede declarar capacidades que habilitan comportamiento adicional durante la configuración. Las capacidades se listan como un array de identificadores de capacidad.

### Bundle Assets (`greentic.cap.bundle_assets.read.v1`)

Cuando la capacidad `greentic.cap.bundle_assets.read.v1` está presente en `bundle.yaml`, `gtc setup` crea un directorio `assets/` en la raíz del bundle y genera archivos de assets desde los provider packs dentro de él.

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

En concreto, los assets de los provider packs (como las entradas `assets/webchat-gui/`) se copian al directorio `assets/` a nivel de bundle. Esto te da un lugar para personalizar los assets del proveedor, por ejemplo, skins de webchat-gui y archivos de configuración, sin modificar el contenido original del pack.
