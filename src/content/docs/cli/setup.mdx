## Overview

**Greentic Setup** is the configuration engine for Greentic bundles. Running `gtc setup <bundle>` opens an interactive web dashboard in your browser where you can configure provider credentials, inspect current scopes, and run the guided setup wizard for each provider — all from a secure, locally-hosted single-page application.

For CI or SSH environments, `gtc setup --no-ui <bundle>` falls back to the terminal dialoguer flow.

## Launching

```bash
# Launch the web dashboard (default)
gtc setup ./my-bundle

# Headless terminal mode (CI / SSH)
gtc setup --no-ui ./my-bundle

# Apply a pre-built answers file without UI
gtc setup --answers answers.json ./my-bundle
```

When you run `gtc setup ./my-bundle`, the binary:

1. Loads and validates your bundle configuration.
2. Starts a local [Axum](https://docs.rs/axum) HTTP server bound to `127.0.0.1` on a random available port.
3. Generates a one-time bearer token stored only in memory — never written to disk.
4. Opens your default browser to `http://127.0.0.1:<port>/?token=<token>`.
5. Serves the Greentic Setup single-page application.

The server shuts down automatically when you close the browser tab or use the in-dashboard **Close** button.

## Dashboard

The Greentic Setup dashboard is a single-page application built with [Alpine.js](https://alpinejs.dev/). It provides a clean, minimal interface for managing your bundle configuration without leaving the browser.

### Layout

The dashboard has three main areas:

- **Topbar** — displays the bundle name and a close button that gracefully shuts down the server.
- **Sidebar** — contains three scope selectors (tenant, environment, team) and the navigation menu.
- **Main content area** — renders the active view.

### Sidebar navigation

Three dropdown selectors at the top of the sidebar let you change the active scope context:

| Selector | What it controls |
|----------|-----------------|
| Tenant | The tenant namespace for secrets and provider config |
| Environment | `demo`, `dev`, `staging`, or a custom value |
| Team | Sub-team scope within the tenant |

The **Manage** nav section shows:

| Item | Status |
|------|--------|
| Overview | Active in Phase 1a |
| Providers | Coming in Phase 1b |
| Secrets | Coming in Phase 1b |
| Capabilities | Coming in Phase 1b |

Items tagged "1b" are read-only placeholders in the current release.

### Overview view

The Overview is the default landing view. It contains:

- **Welcome heading** with the bundle display name.
- **Stat row** showing four summary chips: Scopes configured, Providers found, Secrets stored, and Warnings.
- **Scope list** — one card per configured scope (tenant/env/team combination), each showing the provider chips that have been set up in that scope.
- **Empty state** — if no scopes have been configured yet, an empty-state panel prompts you to add the first tenant using the **Add tenant** button.

## Wizard

Clicking **Add tenant** (or a per-scope **Configure** button) launches the inline setup wizard. The wizard runs inside the dashboard shell — it does not navigate away from the page.

### Wizard flow

The wizard collects configuration in a series of steps:

<Steps>

1. **Select scope** — choose or confirm the tenant, environment, and team.

2. **Choose providers** — toggle which providers you want to enable for this scope.

3. **Configure each provider** — for each enabled provider, fill in the required fields (API keys, public webhook URL, etc.).

4. **Review** — see a summary of all values before they are persisted.

5. **Finish** — credentials are written to the secrets store; the Overview refreshes to show the new scope.

</Steps>

### Field types

The wizard supports the following input types:

| Type | Usage |
|------|-------|
| `text` | Plain string values (URLs, IDs, usernames) |
| `password` | Sensitive values — rendered as masked inputs, cleared from memory on finish |
| `textarea` | Multi-line values (certificate PEM blocks, JSON payloads) |
| `select` | Enumerated options |
| `switch` | Boolean toggles (enable/disable a provider) |

### Navigation

Each step has **Back**, **Continue**, and **Cancel** actions. The final step replaces **Continue** with **Finish**. Validation runs per-step before advancing; errors are shown inline under the relevant field.

## Headless mode

If you are working over SSH, in a CI pipeline, or have disabled browser launch, pass `--no-ui`:

```bash
gtc setup --no-ui ./my-bundle
```

This invokes the legacy terminal dialoguer flow. All the same configuration values are collected interactively at the prompt, and secrets are persisted to the same store as the dashboard wizard.

### Answers files

Answers files let you automate setup without any interactive prompts:

```bash
# Generate a template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Generate a template with optional fields
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle

# Apply answers
gtc setup --answers answers.json ./my-bundle
```

#### Demo catalog answers

Every demo in the [`greentic-demo`](https://github.com/greenticai/greentic-demo) catalog publishes a `*-setup-answers.json` document. Apply it directly:

```bash
gtc setup ./quickstart-demo-bundle \
  --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

See [Running Demos](/getting-started/running-demos/) for the full catalog with descriptions and required credentials.

<Aside type="note">
Greentic Setup auto-detects the bundle's tenant from the `tenants/` directory, so secrets are written to the correct tenant URI even when the bundle was not created with the default `demo` tenant. It also runs **dependency capability validation** across every pack and warns if any required capability (for example `greentic:state/state-store`) is unsatisfied.
</Aside>

## Security

Greentic Setup takes a security-first approach for all local UI sessions:

| Control | Implementation |
|---------|---------------|
| Loopback only | Server binds exclusively to `127.0.0.1` — no network exposure |
| Random port | A random available port is selected at startup — no fixed port to guess |
| Bearer token | One-time token generated in memory, passed via URL query param on first load, then stored in the browser session only |
| Origin enforcement | Every API request is validated against an `Origin` allowlist; cross-origin requests are rejected |
| No secrets in logs | Credential values are never written to stdout, stderr, or log files |
| No secrets on disk | The bearer token and wizard answers are held in memory only; `Zeroize` is called on sensitive structs after use |
| Same-origin assets | All JS, CSS, and font assets are served from the same local server — no external CDN calls |
| No telemetry | The dashboard makes no outbound network calls |

## Bundle capabilities

The `bundle.yaml` file can declare capabilities that unlock additional behavior during setup.

### Bundle assets (`greentic.cap.bundle_assets.read.v1`)

When this capability is declared, Greentic Setup scaffolds an `assets/` directory at the bundle root and copies provider pack assets into it (for example `assets/webchat-gui/`). You can then modify these files freely; bundle-level asset files override pack defaults at runtime.

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

<Aside type="tip">
After setup scaffolds the assets, your bundle-level customizations take precedence over the pack defaults when the runtime loads them.
</Aside>

## Roadmap

Greentic Setup Phase 1a delivers the dashboard shell, Overview view, and inline wizard. Upcoming work in **Phase 1b** includes:

- **Providers** — browse all packs in the bundle, enable or disable individual providers, re-run setup for a single provider without touching others.
- **Secrets** — full CRUD for the secrets store: create, read (masked), update, and delete individual secret entries per scope.
- **Capabilities** — toggle capability flags per pack and inspect the dependency graph.
- **Rebuild trigger** — request a hot reload of the running bundle from the dashboard without restarting the full runtime.

## Troubleshooting

### Browser does not open

If your system does not support automatic browser launch, copy the URL printed to the terminal and paste it into your browser manually:

```
Greentic Setup running at http://127.0.0.1:52341/?token=...
```

### Webhook registration failed

```
Error: Failed to register webhook: 401 Unauthorized
```

Check that your bot token is correct and has not expired.

### Missing required field

```
Error: Missing required field 'bot_token' for messaging-telegram
```

Ensure all required fields are filled in the wizard or present in your answers file.

### Public URL not accessible

```
Error: Webhook verification failed: Connection refused
```

Your public URL must be reachable from the internet. Check that ngrok or cloudflared is running.

### Credential not found

```
Error: Secret 'slack_bot_token' not found
```

Run `gtc setup` to configure the provider, or check your answers file.
