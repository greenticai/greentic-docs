## Overview

The `gtc setup` command configures extension packs in your Greentic bundle. It runs setup flows for enabled messaging, event, OAuth, secrets, and runtime extensions, collecting credentials and configuring webhooks where needed.

## Usage

```bash
gtc setup [OPTIONS] <BUNDLE_PATH>
```

## Options

| Option | Description |
|--------|-------------|
| `--answers <FILE>` | Path to answers file (non-interactive) |
| `--key <KEY>` | Encryption/decryption key for answer documents that include secrets |
| `--dry-run` | Preview setup without making changes |
| `--emit-answers <FILE>` | Generate answers template |
| `--advanced` | Include optional fields in answers template |
| `--tenant <TENANT>` | Tenant identifier (default: `demo`) |
| `--team <TEAM>` | Team identifier |
| `--env <ENV>` | Environment (`dev`, `staging`, `prod`; default: `dev`) |
| `--locale <LOCALE>` | UI locale as a BCP-47 tag |
| `--ui` / `--no-ui` | Enable or disable the web setup UI |
| `--non-interactive` | Fail instead of prompting when answers are incomplete |

## Interactive Setup

Run setup interactively to be prompted for each extension:

```bash
gtc setup ./my-bundle
```

The CLI prompts for:
- Platform setup, such as static route policy, public base URL, deployment target, and local tunnel mode
- Extension credentials, such as API keys, tokens, and OAuth client IDs
- Provider-specific setup actions, such as OAuth install buttons and device-code login
- Optional configuration options when `--advanced` is enabled

## Non-Interactive Setup

### Demo Catalog Flows

Every demo in the [`greentic-demo`](https://github.com/greenticai/greentic-demo) catalog publishes a `*-setup-answers.json` document alongside its pack. Apply it to a bundle that you created with `gtc wizard`:

```bash
gtc setup ./deep-research-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/deep-research-demo-setup-answers.json
```

The same pattern works for any demo — just substitute the demo ID. See [Running Demos](/getting-started/running-demos/) for the full catalog with descriptions and required credentials.

You can also point `gtc setup` directly at a published `.gtbundle` artifact:

```bash
gtc setup https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo.gtbundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo-setup-answers.json
```

<Aside type="note">
`gtc setup` auto-detects the bundle's tenant from the `tenants/` directory, so secrets are written to the correct tenant URI even when the bundle was not created with the default `demo` tenant. It also runs **dependency capability validation** across every pack in the bundle and warns if any required capability (e.g. `greentic:state/state-store`) is still unsatisfied.
</Aside>

### Generate Answers Template

First, generate a template with all required fields:

```bash
# Basic template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Full template (including optional fields)
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle
```

### Answers File Format

```json title="answers.json"
{
  "greentic_setup_version": "1.0.0",
  "bundle_source": "./my-bundle",
  "tenant": "demo",
  "team": null,
  "env": "dev",
  "platform_setup": {
    "static_routes": {
      "public_web_enabled": true,
      "public_base_url": "https://xxx.trycloudflare.com",
      "public_surface_policy": "enabled",
      "default_route_prefix_policy": "pack_declared",
      "tenant_path_policy": "pack_declared"
    },
    "deployment_targets": [],
    "tunnel": {
      "mode": "cloudflared"
    }
  },
  "setup_answers": {
    "messaging-telegram": {
      "public_base_url": "https://xxx.trycloudflare.com",
      "api_base_url": "https://api.telegram.org",
      "telegram_bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
    },
    "messaging-slack": {
      "slack_configuration_access_token": "xoxe.xoxp-xxx",
      "slack_configuration_refresh_token": "xoxe-1-xxx"
    },
    "messaging-teams": {
      "client_id": "6c115a7a-f656-49c4-975a-5e831efae833",
      "desired_channel_name": "hr onboarding"
    },
    "messaging-webex": {
      "webex_bot_token": "your-webex-bot-access-token"
    },
    "messaging-webchat-gui": {
      "public_base_url": "https://xxx.trycloudflare.com",
      "mode": "websocket",
      "route": "webchat"
    }
  }
}
```

Generated answers files are structured documents. Provider answers live under `setup_answers`, keyed by extension id. `platform_setup` stores bundle-level setup choices such as public static routes, deployment targets, and tunnel mode. Older flat files keyed directly by provider id still load, but new docs and generated templates use the structured format.

Generated secrets are not prompted when the provider can create them itself. For example, Webex generates its webhook signing secret and WebChat GUI generates its JWT signing key during setup.

### Apply Answers

```bash
gtc setup --answers answers.json ./my-bundle
```

## Extension-Specific Configuration

### Telegram

```json
{
  "setup_answers": {
    "messaging-telegram": {
      "public_base_url": "https://your-domain.com",
      "api_base_url": "https://api.telegram.org",
      "telegram_bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
    }
  }
}
```

`api_base_url` is required by the current pack schema. Use `https://api.telegram.org` unless you run a compatible Telegram Bot API endpoint.

### Slack

```json
{
  "setup_answers": {
    "messaging-slack": {
      "slack_configuration_access_token": "xoxe.xoxp-xxx",
      "slack_configuration_refresh_token": "xoxe-1-xxx"
    }
  }
}
```

Slack setup uses the provider's setup action to register or update the Slack app, stores `slack_app_id`, and persists the configuration access and refresh tokens for future manifest updates.

### Microsoft Teams

```json
{
  "setup_answers": {
    "messaging-teams": {
      "client_id": "6c115a7a-f656-49c4-975a-5e831efae833",
      "desired_channel_name": "hr onboarding"
    }
  }
}
```

Teams setup is Graph-first by default. It uses device-code OAuth, discovers joined teams and channels, stores both IDs and display names, and can create the desired channel when it is missing and the signed-in user has `Channel.Create` permission.

### WhatsApp (via Meta)

```json
{
  "setup_answers": {
    "messaging-whatsapp": {
      "phone_number_id": "123456789",
      "business_account_id": "987654321",
      "api_base_url": "https://graph.facebook.com",
      "api_version": "v17.0",
      "whatsapp_token": "EAAxxxxx",
      "whatsapp_verify_token": "your-verify-token",
      "public_base_url": "https://your-domain.com"
    }
  }
}
```

### Webex

```json
{
  "setup_answers": {
    "messaging-webex": {
      "webex_bot_token": "your-webex-bot-access-token"
    }
  }
}
```

Webex setup only asks for the bot access token. The provider generates the webhook signing secret, discovers the bot identity, and registers webhooks for direct 1:1 messages, room mentions, and Adaptive Card actions.

### WebChat

Use `messaging-webchat-gui` for the hosted browser UI and embeddable widget:

```json
{
  "setup_answers": {
    "messaging-webchat-gui": {
      "public_base_url": "https://your-domain.com",
      "mode": "websocket",
      "route": "webchat"
    }
  }
}
```

WebChat GUI defaults to WebSocket mode. Setup generates the JWT signing key automatically and persists it as a secret.

Use `messaging-webchat` only when you need the Direct Line-compatible provider without the hosted GUI:

```json
{
  "setup_answers": {
    "messaging-webchat": {
      "public_base_url": "https://your-domain.com",
      "mode": "directline",
      "route": "webchat",
      "jwt_signing_key": "replace-with-a-random-secret"
    }
  }
}
```

### Email

```json
{
  "setup_answers": {
    "messaging-email": {
      "host": "smtp.example.com",
      "port": 587,
      "username": "mailer@example.com",
      "password": "your-smtp-password",
      "from_address": "bot@example.com",
      "tls_mode": "starttls"
    }
  }
}
```

`messaging-email` is SMTP. `messaging-microsoft-email` uses Microsoft Graph mail and expects Graph tenant, client, client secret, refresh token, and from-address answers.

## Public URL Configuration

<Aside type="caution">
Webhook-based extensions require a publicly accessible URL. The `public_base_url` will be automatically replaced when using ngrok or cloudflared with `gtc start`.
</Aside>

### Using ngrok

```bash
# Start ngrok in separate terminal
ngrok http 8080

# Copy the HTTPS URL (e.g., https://abc123.ngrok-free.app)
# Use this as public_base_url in answers
```

### Using Cloudflared

```bash
# Cloudflared generates URL automatically with gtc start
gtc start ./my-bundle --cloudflared on
```

## Bundle Capabilities

The `bundle.yaml` file can declare capabilities that unlock additional behavior during setup. Capabilities are listed as an array of capability identifiers.

### Bundle Assets (`greentic.cap.bundle_assets.read.v1`)

When the `greentic.cap.bundle_assets.read.v1` capability is present in `bundle.yaml`, `gtc setup` creates an `assets/` directory at the bundle root and scaffolds eligible asset files from extension packs into it.

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

Specifically, public assets from extension packs (currently `assets/webchat-gui/` entries) are copied into the bundle-level `assets/` directory. This gives you a place to customize extension assets -- for example, webchat-gui skins and configuration files -- without modifying the original pack contents.

**How it works:**

- During setup, extension packs that include eligible asset directories (e.g., `assets/webchat-gui/`) have those files scaffolded into the bundle root `assets/` directory.
- You can then edit these files freely. Bundle-level asset files override pack defaults at runtime.
- This is useful for customizing the look and feel of webchat widgets, adjusting configuration, or applying branded skins.

<Aside type="tip">
After setup scaffolds the assets, you can safely modify them. Your bundle-level customizations take precedence over the pack defaults when the runtime loads assets.
</Aside>

## Setup Flow Execution

When you run `gtc setup`, it:

<Steps>

1. **Loads bundle configuration**

   Reads `bundle.yaml` and the resolved bundle metadata to find extension packs and setup requirements.

2. **Validates answers**

   Checks all required fields are present.

3. **Runs setup flows**

   Executes each extension's setup flow (e.g., webhook registration).

4. **Stores credentials**

   Saves credentials to the configured secrets store.

5. **Runs verification**

   Executes `verify_flow` to confirm setup worked.

</Steps>

## Re-run Setup

`gtc setup` is idempotent. Re-run it with the same answers file after credential or public URL changes:

```bash
gtc setup ./my-bundle --answers answers.json
```

For headless automation, combine an answers file with strict non-interactive mode:

```bash
gtc setup ./my-bundle --answers answers.json --non-interactive --no-ui
```

## Troubleshooting

### Webhook Registration Failed

```
Error: Failed to register webhook: 401 Unauthorized
```

Check that your bot token is correct and not expired.

### Missing Required Field

```
Error: missing required setup answer for telegram_bot_token
```

Ensure all required fields are in your answers file.

### Public URL Not Accessible

```
Error: Webhook verification failed: Connection refused
```

Ensure your public URL is accessible from the internet. Check ngrok/cloudflared is running.

### Credential Not Found

```
Error: Secret 'SLACK_CONFIGURATION_ACCESS_TOKEN' not found
```

Run `gtc setup` to configure the extension, or check your answers file and secrets store.

## Best Practices

1. **Use answers files in CI/CD** - Avoid interactive prompts in automated pipelines
2. **Keep credentials secure** - Don't commit answers files with real tokens to git
3. **Use environment variables** - Reference secrets via `$ENV_VAR` in answers
4. **Test with dry-run** - Preview setup before making changes
5. **Regenerate on URL change** - Re-run setup when your public URL changes
