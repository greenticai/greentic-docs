## 概要

**Greentic Setup** は Greentic bundle の設定エンジンです。`gtc setup <bundle>` を実行すると、ブラウザにインタラクティブな Web ダッシュボードが開き、provider の認証情報の設定、現在のスコープの確認、各 provider の設定ウィザードの実行を、セキュアなローカルホスト型シングルページアプリケーションから行えます。

CI または SSH 環境では、`gtc setup --no-ui <bundle>` を使用すると端末のダイアローガーフローにフォールバックします。

## 起動

```bash
# Web ダッシュボードを起動（デフォルト）
gtc setup ./my-bundle

# ヘッドレス端末モード（CI / SSH）
gtc setup --no-ui ./my-bundle

# UI なしで事前構築済み回答ファイルを適用
gtc setup --answers answers.json ./my-bundle
```

`gtc setup ./my-bundle` を実行すると、バイナリは以下の処理を行います：

1. bundle 設定を読み込み、検証します。
2. ランダムな使用可能ポートで `127.0.0.1` にバインドされたローカル [Axum](https://docs.rs/axum) HTTP サーバーを起動します。
3. メモリのみに保存されるワンタイムベアラートークンを生成します（ディスクへの書き込みは行いません）。
4. `http://127.0.0.1:<port>/?token=<token>` をデフォルトブラウザで開きます。
5. Greentic Setup シングルページアプリケーションを提供します。

ブラウザタブを閉じるか、ダッシュボードの **Close** ボタンを使用すると、サーバーは自動的にシャットダウンします。

## ダッシュボード

Greentic Setup ダッシュボードは [Alpine.js](https://alpinejs.dev/) で構築されたシングルページアプリケーションです。ブラウザを離れることなく bundle 設定を管理するためのクリーンでミニマルなインターフェースを提供します。

### レイアウト

ダッシュボードには 3 つの主要エリアがあります：

- **トップバー** — bundle 名とサーバーをグレースフルにシャットダウンする閉じるボタンを表示します。
- **サイドバー** — 3 つのスコープセレクター（テナント、環境、チーム）とナビゲーションメニューが含まれます。
- **メインコンテンツエリア** — アクティブなビューをレンダリングします。

### サイドバーナビゲーション

サイドバー上部の 3 つのドロップダウンセレクターで、アクティブなスコープコンテキストを変更できます：

| セレクター | 制御内容 |
|------------|----------|
| テナント | シークレットと provider 設定のテナント名前空間 |
| 環境 | `demo`、`dev`、`staging`、またはカスタム値 |
| チーム | テナント内のサブチームスコープ |

**Manage** ナビセクションには以下が表示されます：

| 項目 | ステータス |
|------|-----------|
| Overview | Phase 1a でアクティブ |
| Providers | Phase 1b で提供予定 |
| Secrets | Phase 1b で提供予定 |
| Capabilities | Phase 1b で提供予定 |

「1b」タグの付いた項目は現在のリリースでは読み取り専用のプレースホルダーです。

### Overview ビュー

Overview はデフォルトのランディングビューです。以下が含まれます：

- bundle 表示名を含む**ウェルカム見出し**。
- 設定済みスコープ数、検出された provider 数、保存されたシークレット数、警告数の 4 つのサマリーチップを示す**統計行**。
- **スコープリスト** — 設定済みスコープ（テナント/環境/チームの組み合わせ）ごとのカード。そのスコープで設定された provider チップを表示します。
- **空の状態** — スコープがまだ設定されていない場合、空の状態パネルが表示され、**Add tenant** ボタンで最初のテナントを追加するよう促します。

## ウィザード

**Add tenant** ボタン（またはスコープごとの **Configure** ボタン）をクリックすると、インライン設定ウィザードが起動します。ウィザードはダッシュボードシェル内で実行されます（別ページには遷移しません）。

### ウィザードフロー

ウィザードは一連のステップで設定を収集します：

<Steps>

1. **スコープを選択** — テナント、環境、チームを選択または確認します。

2. **provider を選択** — このスコープで有効にする provider を切り替えます。

3. **各 provider を設定** — 有効にした各 provider に必要なフィールド（API キー、パブリック Webhook URL など）を入力します。

4. **確認** — 値が保存される前にすべての値のサマリーを確認します。

5. **完了** — 認証情報がシークレットストアに書き込まれ、Overview が更新されて新しいスコープが表示されます。

</Steps>

### フィールドタイプ

ウィザードは以下の入力タイプをサポートします：

| タイプ | 用途 |
|--------|------|
| `text` | プレーンな文字列値（URL、ID、ユーザー名） |
| `password` | 機密値 — マスクされた入力でレンダリングされ、完了後にメモリからクリアされます |
| `textarea` | 複数行の値（証明書 PEM ブロック、JSON ペイロード） |
| `select` | 列挙されたオプション |
| `switch` | ブール値トグル（provider の有効/無効） |

### ナビゲーション

各ステップには **Back**、**Continue**、**Cancel** アクションがあります。最後のステップでは **Continue** が **Finish** に置き換わります。検証はステップごとに実行され、エラーは関連するフィールドの下にインラインで表示されます。

## ヘッドレスモード

SSH 経由での作業中、CI パイプライン内、またはブラウザ起動が無効な場合は、`--no-ui` を渡します：

```bash
gtc setup --no-ui ./my-bundle
```

これにより、レガシー端末ダイアローガーフローが呼び出されます。同じ設定値がプロンプトでインタラクティブに収集され、シークレットはダッシュボードウィザードと同じストアに保存されます。

### 回答ファイル

回答ファイルを使用すると、インタラクティブなプロンプトなしでセットアップを自動化できます：

```bash
# テンプレートを生成（必須フィールドのみ）
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# オプションフィールドを含むテンプレートを生成
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle

# 回答を適用
gtc setup --answers answers.json ./my-bundle
```

#### デモカタログの回答

[`greentic-demo`](https://github.com/greenticai/greentic-demo) カタログの各デモには `*-setup-answers.json` ドキュメントが公開されています。直接適用できます：

```bash
gtc setup ./quickstart-demo-bundle \
  --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

説明と必要な認証情報を含む完全なカタログは[デモの実行](/ja/getting-started/running-demos/)を参照してください。

<Aside type="note">
Greentic Setup は `tenants/` ディレクトリから bundle のテナントを自動検出するため、bundle がデフォルトの `demo` テナントで作成されていない場合でも、シークレットは正しいテナント URI に書き込まれます。また、bundle 内のすべてのパックに対して**依存関係の機能検証**を実行し、必要な機能（例：`greentic:state/state-store`）が満たされていない場合に警告を出します。
</Aside>

## セキュリティ

Greentic Setup はすべてのローカル UI セッションに対してセキュリティファーストのアプローチを取っています：

| 制御 | 実装 |
|------|------|
| ループバックのみ | サーバーは `127.0.0.1` にのみバインド — ネットワークへの露出なし |
| ランダムポート | 起動時にランダムな使用可能ポートを選択 — 固定ポートへの推測攻撃なし |
| ベアラートークン | メモリで生成されるワンタイムトークン。最初のロード時に URL クエリパラメータで渡され、その後はブラウザセッションにのみ保存されます |
| Origin 強制 | すべての API リクエストは `Origin` 許可リストに対して検証されます。クロスオリジンリクエストは拒否されます |
| ログにシークレットなし | 認証情報の値は stdout、stderr、ログファイルに書き込まれません |
| ディスクにシークレットなし | ベアラートークンとウィザードの回答はメモリのみに保持され、使用後に機密構造体に対して `Zeroize` が呼び出されます |
| 同一オリジンアセット | すべての JS、CSS、フォントアセットは同じローカルサーバーから提供されます — 外部 CDN 呼び出しなし |
| テレメトリなし | ダッシュボードはアウトバウンドネットワーク呼び出しを行いません |

## Bundle の機能

`bundle.yaml` ファイルでは、セットアップ中に追加の動作を有効にする機能を宣言できます。

### Bundle アセット（`greentic.cap.bundle_assets.read.v1`）

この機能が宣言されると、Greentic Setup は bundle ルートに `assets/` ディレクトリを作成し、provider パックアセット（例：`assets/webchat-gui/`）をそこにコピーします。その後これらのファイルを自由に変更でき、bundle レベルのアセットファイルは実行時にパックのデフォルトをオーバーライドします。

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

<Aside type="tip">
セットアップがアセットを作成した後、bundle レベルのカスタマイズはランタイムがアセットを読み込む際にパックのデフォルトより優先されます。
</Aside>

## ロードマップ

Greentic Setup Phase 1a はダッシュボードシェル、Overview ビュー、インラインウィザードを提供します。**Phase 1b** での今後の作業には以下が含まれます：

- **Providers** — bundle 内のすべてのパックを参照し、個々の provider を有効または無効にし、他の provider に影響を与えることなく単一の provider のセットアップを再実行します。
- **Secrets** — シークレットストアの完全な CRUD：スコープごとの個々のシークレットエントリの作成、読み取り（マスク済み）、更新、削除。
- **Capabilities** — パックごとの機能フラグの切り替えと依存関係グラフの確認。
- **Rebuild トリガー** — 完全なランタイムの再起動なしにダッシュボードから実行中の bundle のホットリロードをリクエストします。

## トラブルシューティング

### ブラウザが開かない

システムが自動ブラウザ起動をサポートしていない場合は、端末に表示された URL をコピーしてブラウザに手動で貼り付けてください：

```
Greentic Setup running at http://127.0.0.1:52341/?token=...
```

### Webhook 登録に失敗した

```
Error: Failed to register webhook: 401 Unauthorized
```

ボットトークンが正しく、有効期限が切れていないことを確認してください。

### 必須フィールドが不足している

```
Error: Missing required field 'bot_token' for messaging-telegram
```

すべての必須フィールドがウィザードで入力されているか、回答ファイルに存在することを確認してください。

### パブリック URL にアクセスできない

```
Error: Webhook verification failed: Connection refused
```

パブリック URL はインターネットから到達可能である必要があります。ngrok または cloudflared が実行中であることを確認してください。

### 認証情報が見つからない

```
Error: Secret 'slack_bot_token' not found
```

`gtc setup` を実行して provider を設定するか、回答ファイルを確認してください。
