## 概要

`gtc setup` コマンドは Greentic bundle 内の provider を設定します。有効になっている各 provider の setup flow を実行し、認証情報を収集して webhook を設定します。

## 使い方

```bash
gtc setup [OPTIONS] <BUNDLE_PATH>
```

## オプション

| Option | 説明 |
|--------|-------------|
| `--answers <FILE>` | answers file のパス（非対話モード） |
| `--dry-run` | 変更を加えずに setup をプレビューする |
| `--emit-answers <FILE>` | answers テンプレートを生成する |
| `--advanced` | answers テンプレートに任意フィールドを含める |
| `--provider <NAME>` | 特定の provider のみ setup する |
| `--skip <NAME>` | 特定の provider をスキップする |
| `-v, --verbose` | verbose 出力を有効にする |

## 対話型セットアップ

各 provider について入力を求められる対話モードで setup を実行します:

```bash
gtc setup ./my-bundle
```

CLI は以下を尋ねます:
- provider の認証情報（API keys、tokens）
- webhook 用の公開 URL
- 任意の設定オプション

## 非対話型セットアップ

### Demo Catalog Flows

[`greentic-demo`](https://github.com/greenticai/greentic-demo) catalog 内の各 demo は、pack と一緒に `*-setup-answers.json` ドキュメントを公開しています。これを `gtc wizard` で作成した bundle に適用します:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

同じパターンはどの demo に対しても使えます。demo ID を置き換えるだけです。説明と必要な認証情報を含む完全な catalog は [Running Demos](/ja/getting-started/running-demos/) を参照してください。

`gtc setup` を公開済みの `.gtbundle` artifact に直接向けることもできます:

```bash
gtc setup https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo.gtbundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo-setup-answers.json
```

<Aside type="note">
`gtc setup` は `tenants/` directory から bundle の tenant を自動検出するため、bundle がデフォルトの `demo` tenant で作成されていなくても、secret は正しい tenant URI に書き込まれます。また、bundle 内のすべての pack に対して **dependency capability validation** を実行し、必要な capability（例: `greentic:state/state-store`）がまだ満たされていない場合は警告します。
</Aside>

### Answers Template を生成する

まず、必須フィールドをすべて含むテンプレートを生成します:

```bash
# Basic template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Full template (including optional fields)
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle
```

## Provider 固有の設定

### Telegram

```json
{
  "messaging-telegram": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
  }
}
```

**認証情報の取得方法:**
1. Telegram で [@BotFather](https://t.me/botfather) にメッセージを送る
2. `/newbot` を送信して案内に従う
3. bot token をコピーする

### Slack

```json
{
  "messaging-slack": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "xoxb-xxx-xxx-xxx",
    "slack_app_id": "A07XXXXXX",
    "slack_configuration_token": "xoxe.xoxp-xxx"
  }
}
```

**認証情報の取得方法:**
1. [api.slack.com/apps](https://api.slack.com/apps) にアクセスする
2. 新しい app を作成するか既存のものを選択する
3. OAuth & Permissions から Bot Token を取得する
4. Basic Information から App ID を取得する
5. App Manifest から Configuration Token を取得する

### Microsoft Teams

```json
{
  "messaging-teams": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "app_id": "your-app-id",
    "app_password": "your-app-password",
    "tenant_id": "your-tenant-id"
  }
}
```

### WhatsApp (via Meta)

```json

  "messaging-whatsapp": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "phone_number_id": "123456789",

  }
}
```

## Public URL Configuration

<Aside type="caution">
Webhook ベースの provider には外部からアクセス可能な URL が必要です。`public_base_url` は、`gtc start` で ngrok または cloudflared を使う場合に自動的に置き換えられます。
</Aside>

### ngrok を使う

```bash
# Start ngrok in separate terminal
ngrok http 8080

# Copy the HTTPS URL (e.g., https://abc123.ngrok-free.app)
# Use this as public_base_url in answers
```

### Cloudflared を使う

```bash
# Cloudflared generates URL automatically with gtc start
gtc start ./my-bundle --cloudflared on
```

## Bundle Capabilities

`bundle.yaml` file では、setup 中に追加の挙動を有効にする capabilities を宣言できます。Capabilities は capability identifier の配列として列挙されます。

### Bundle Assets (`greentic.cap.bundle_assets.read.v1`)

`bundle.yaml` に `greentic.cap.bundle_assets.read.v1` capability が存在する場合、`gtc setup` は bundle ルートに `assets/` directory を作成し、provider pack から asset file をそこへ scaffold します。

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

具体的には、provider pack の assets（`assets/webchat-gui/` entries など）が bundle レベルの `assets/` directory にコピーされます。これにより、元の pack 内容を変更せずに、webchat-gui の skin や設定ファイルなどの provider assets をカスタマイズできます。

**仕組み:**

- setup 中、asset directory を含む provider pack（例: `assets/webchat-gui/`）のファイルが、bundle ルートの `assets/` directory に scaffold されます。
- その後はこれらの file を自由に編集できます。bundle レベルの asset file は runtime 時に pack のデフォルトを上書きします。
- これは、webchat widget の look and feel の調整、設定の変更、brand に合わせた skin の適用に便利です。

<Aside type="tip">
setup により assets が scaffold された後は、安全に変更できます。runtime が assets を読み込む際、bundle レベルのカスタマイズが pack のデフォルトより優先されます。
</Aside>

## Setup Flow Execution

`gtc setup` を実行すると、次の処理を行います:

<Steps>

1. **bundle 設定を読み込む**

   `greentic.demo.yaml` を読み込み、provider を見つけます。

2. **answers を検証する**

   必須フィールドがすべて存在することを確認します。

3. **setup flows を実行する**

   各 provider の `setup_flow` を実行します（例: webhook registration）。

4. **認証情報を保存する**

   認証情報を設定済みの secrets store に保存します。

5. **検証を実行する**

   `verify_flow` を実行して setup が正しく機能したことを確認します。

</Steps>

## Selective Setup

### 単一 Provider を setup する

```bash
gtc setup --provider messaging-telegram ./my-bundle
```

### 特定の Provider をスキップする

```bash
gtc setup --skip messaging-teams ./my-bundle
```

### Setup を再実行する

```bash
# Force re-setup (useful after URL changes)
gtc start ./my-bundle --force-setup
```

## Troubleshooting

### Webhook Registration Failed

```
Error: Failed to register webhook: 401 Unauthorized
```

bot token が正しく、有効期限切れでないことを確認してください。

### Missing Required Field

```
Error: Missing required field 'bot_token' for messaging-telegram
```

answers file に必須フィールドがすべて含まれていることを確認してください。

### Public URL Not Accessible

```
Error: Webhook verification failed: Connection refused
```

公開 URL にインターネットからアクセスできることを確認してください。ngrok/cloudflared が実行中か確認します。

### Credential Not Found

```
Error: Secret 'slack_bot_token' not found
```

provider を設定するために `gtc setup` を実行するか、answers file を確認してください。

## Best Practices

1. **CI/CD では answers file を使う** - 自動化パイプラインでは対話プロンプトを避ける
2. **認証情報を安全に保つ** - 実際の token を含む answers file を git に commit しない
3. **environment variables を使う** - answers 内では `$ENV_VAR` で secret を参照する
4. **dry-run でテストする** - 変更を加える前に setup をプレビューする
5. **URL が変わったら再生成する** - 公開 URL が変わったら setup を再実行する
