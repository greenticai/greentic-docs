## 概要

`gtc wizard` コマンドは wizard answers から新しい Greentic bundle を作成します。設定ファイル、provider のセットアップ、app templates を含む bundle 全体の構造を scaffold します。

## 使い方

```bash
gtc wizard [OPTIONS]
```

## オプション

| Option | 説明 |
|--------|-------------|
| `--answers <FILE \| URL>` | wizard answers file のパスまたは URL（JSON または YAML）。ローカルパスとリモートの `http://` / `https://` URLs をサポートします。 |
| `--dry-run` | 書き込まずに生成される file をプレビューする |
| `--output <DIR>` | 出力 directory（デフォルト: current directory） |
| `--template <NAME>` | 特定の template を使う |
| `-v, --verbose` | verbose 出力を有効にする |

## Wizard Answers File

### JSON Format

```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0",
    "description": "Customer support digital worker"
  },
  "providers": {
    "messaging": ["telegram", "slack"],
    "events": ["webhook", "timer"]
  },
  "apps": [
    {
      "name": "support-bot",
      "template": "customer-service"
    }
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [
        {
          "id": "default",
          "channels": ["telegram", "slack"]
        }
      ]
    }
  ]
}
```

### YAML Format

```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"
  description: Customer support digital worker

providers:
  messaging:
    - telegram
    - slack
  events:
    - webhook
    - timer

apps:
  - name: support-bot
    template: customer-service

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - telegram
          - slack
```

### Capabilities

wizard answers file の `capabilities` 配列は、bundle 作成時に有効になる bundle レベルの機能を制御します。Capabilities によって、標準的な bundle 構造に加えて wizard がどの追加 scaffolding を行うかが決まります。

```json title="create-answers.json"
{
  "capabilities": ["greentic.cap.bundle_assets.read.v1"]
}
```

**利用可能な capabilities:**

| Capability | 説明 |
|------------|-------------|
| `greentic.cap.bundle_assets.read.v1` | bundle レベルの asset overrides を有効にします。存在する場合、wizard は provider packs から webchat-gui skin assets を bundle の `assets/` directory に自動で scaffold します。 |

`greentic.cap.bundle_assets.read.v1` を含めると、生成された bundle には provider packs からコピーされた file（例: webchat-gui skins や configuration）を持つ `assets/` directory が含まれます。これらの bundle レベル file はカスタマイズ可能で、runtime 時に pack defaults を上書きします。

<Aside type="tip">
`capabilities` 配列は `bundle.yaml` に書き込まれ、後から bundle を設定するときに `gtc setup` でも考慮されます。runtime 時の挙動については [gtc setup capabilities section](/ja/cli/setup/#bundle-capabilities) を参照してください。
</Aside>

## Generated Structure

wizard を実行すると、以下が生成されます:

```
my-digital-worker/
├── greentic.demo.yaml          # Main configuration
├── providers/
│   ├── messaging/
│   │   ├── messaging-telegram.gtpack
│   │   └── messaging-slack.gtpack
│   └── events/
│       ├── events-webhook.gtpack
│       └── events-timer.gtpack
├── apps/
│   └── support-bot/
│       ├── app.yaml
│       └── flows/
│           └── on_message.ygtc
├── tenants/
│   └── demo/
│       ├── tenant.gmap
│       └── teams/
│           └── default/
│               └── team.gmap
└── seeds.yaml
```

## Examples

### Demo Catalog Bundles

[`greentic-demo`](https://github.com/greenticai/greentic-demo) repository は、各 demo に対して `*-create-answers.json` と `*-setup-answers.json` を release artifacts として公開しています。これらを直接使って、あとから継続して編集できるローカル bundle を作成できます:

```bash
gtc wizard --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-create-answers.json
```

その後、setup と start を続けます:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
gtc start ./quickstart-demo-bundle
```

公開されている任意の demo（`quickstart`, `quickstart-event`, `hr-onboarding`, `helpdesk-itsm`, `sales-crm`, `supply-chain`, `incident`, `redbutton`, `cloud-deploy-demo`, `weather-mcp-demo`, ...）に対して demo ID を置き換えてください。説明と必要な認証情報を含む完全な catalog は [Running Demos](/ja/getting-started/running-demos/) ページにあります。

<Aside type="tip">
wizard は **capability-based dependency resolution** を実行します。demo の create-answers document が `required_capabilities` を宣言する provider を参照している場合（例: messaging providers が `greentic:state/state-store` を必要とするケース）、wizard は `state-memory` のような要件を満たす pack を自動的に生成 bundle に追加します。手動で列挙する必要はありません。
</Aside>

### Basic Usage

```bash
# Interactive mode
gtc wizard

# With answers file
gtc wizard --answers wizard-answers.yaml
```

### Preview Mode

```bash
# See what would be generated
gtc wizard --answers wizard-answers.yaml --dry-run
```

### Remote Answers File

`--answers` フラグは `http://` と `https://` URLs を受け付けるため、共有 answer documents を中央の場所に保存し、CI/CD pipelines から参照するのが容易です。

```bash
# Fetch answers from a remote URL
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml

# Combine with dry-run to preview before generating
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml --dry-run
```

<Aside type="tip">
remote URL のサポートは、wizard answers を共有 repository や artifact server に保存し、build 時に取得する CI/CD pipelines で有用です。
</Aside>

### Custom Output Directory

```bash
gtc wizard --answers wizard-answers.yaml --output ./my-project
```

### Using Templates

```bash
# List available templates
gtc wizard --list-templates

# Use specific template
gtc wizard --template customer-service --answers answers.yaml
```

## Available Templates

| Template | 説明 |
|----------|-------------|
| `minimal` | 1 つの provider を持つ最小構成の bundle |
| `customer-service` | escalation 付き FAQ bot |
| `helpdesk` | ticket integration を備えた IT support |
| `multi-channel` | 複数の messaging channels |

## Bundle Capabilities

wizard は追加機能を有効にする任意の bundle capabilities を尋ねます。各 capability は、生成された bundle 内の特定の asset と挙動を有効にします。

### Interactive Prompts

`gtc wizard` を対話モードで実行すると、次のように表示されます:

```
? Enable bundle-level assets (./assets/)? [Y/n]
? Enable OAuth login for WebChat GUI? [y/N]
? Enable multi-language for WebChat GUI? [y/N]
? Enable embeddable WebChat widget? [y/N]
```

### Capability Reference

| Prompt | Capability ID | 役割 |
|--------|---------------|-------------|
| Enable bundle-level assets | `greentic.cap.bundle_assets.read.v1` | `./assets/` directory を作成し、provider packs からデフォルト skin files を tenant ごとの directories に scaffold します |
| Enable OAuth login for WebChat GUI | `greentic.cap.webchat.oauth.v1` | tenant config files に OAuth/OIDC authentication provider configuration を追加します |
| Enable multi-language for WebChat GUI | `greentic.cap.webchat.i18n.v1` | WebChat の locale picker を有効化し、`./assets/` 配下に locale JSON files を scaffold します |
| Enable embeddable WebChat widget | `greentic.cap.webchat.embed.v1` | tenant ごとに embed HTML snippets（fullpage link、inline iframe、chat bubble）を生成します |

### Answers File で Capabilities を設定する

対話プロンプトに答える代わりに、`capabilities` 配列を追加して answers file に直接 capabilities を設定できます:

<Tabs>
  <TabItem label="JSON">
```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0"
  },
  "providers": {
    "messaging": ["webchat", "telegram"]
  },
  "capabilities": [
    "greentic.cap.bundle_assets.read.v1",
    "greentic.cap.webchat.oauth.v1",
    "greentic.cap.webchat.i18n.v1",
    "greentic.cap.webchat.embed.v1"
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [{ "id": "default", "channels": ["webchat"] }]
    }
  ]
}
```
  </TabItem>
  <TabItem label="YAML">
```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"

providers:
  messaging:
    - webchat
    - telegram

capabilities:
  - greentic.cap.bundle_assets.read.v1
  - greentic.cap.webchat.oauth.v1
  - greentic.cap.webchat.i18n.v1
  - greentic.cap.webchat.embed.v1

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - webchat
```
  </TabItem>
</Tabs>

<Aside type="tip">
Capabilities は加算式です。`greentic.cap.webchat.embed.v1` を有効にすると、embed snippets は `./assets/` directory に書き込まれるため、暗黙的に `greentic.cap.bundle_assets.read.v1` も必要になります。wizard はこの依存関係を自動的に処理します。
</Aside>

capabilities、scaffold の挙動、overlay mechanism の詳細は [Bundle Assets](/ja/concepts/bundle-assets/) を参照してください。

## Workflow

<Steps>

1. **answers file を作成する**

   JSON または YAML で project configuration を定義します。

2. **wizard を実行する**

   ```bash
   gtc wizard --answers wizard-answers.yaml
   ```

3. **生成された files を確認する**

   生成された bundle 構造を確認します。

4. **provider を設定する**

   ```bash
   gtc setup ./my-digital-worker
   ```

5. **runtime を起動する**

   ```bash
   gtc start ./my-digital-worker
   ```

</Steps>

## Generated Flows のカスタマイズ

wizard は基本的な flow templates を生成します。生成後にカスタマイズしてください:

```yaml title="apps/support-bot/flows/on_message.ygtc"
name: on_message
version: "1.0"
description: Handle incoming messages

nodes:
  # Add your custom nodes here
  - id: analyze
    type: llm
    config:
      model: "gpt-4"
      prompt: "Analyze: {{message}}"
    next: respond

  - id: respond
    type: reply
    config:
      message: "{{analysis_result}}"

triggers:
  - type: message
    target: analyze
```

<Aside type="tip">
disk に書き出す前に、まず `--dry-run` を使って生成される構造をプレビューしてください。
</Aside>

## Component scaffolding: `http_client` capability

When scaffolding a new WASM component (for example via `greentic-component new`), the wizard can mark the component as needing outbound HTTP by setting `http_client: true`. This writes the capability declaration into the component manifest.

<Aside type="caution">
**Desktop runner HTTP gate.** Declaring `http_client: true` is a capability _request_ — the runtime must grant it before the component can actually open outbound sockets. On the **desktop runner profile**, honouring this manifest flag has been a no-op until the HTTP gate fix lands: components that declared `http_client: true` had no `wasi:http/outgoing-handler` implementation linked, and runtime HTTP calls failed.

As of Greentic runner `N.N.N`, the desktop profile honours the manifest and grants `wasi:http/outgoing-handler@0.2.0` automatically when `http_client: true` is declared. {/* TODO: confirm runner version once Bug 2 PR merges */}

**Until the fix is released, the workaround is** to ship your component with a custom WIT world that imports `wasi:http/outgoing-handler@0.2.0` directly, rather than relying on the scaffolded manifest flag. Once Bug 2 is merged and you are on a runner that includes it, you can drop the custom world and use `http_client: true` as intended.

For components that emit Adaptive Cards via HTTP fetches (e.g. RAG lookups before returning an `attachments[]` array), see the [WebChat provider's Attachment envelope shape](/providers/messaging/webchat/#attachment-envelope-shape) section for the expected component-output layout.
</Aside>

## Troubleshooting

### Invalid Answers File

```
Error: Failed to parse answers file
```

JSON/YAML が有効であることを確認してください。syntax を確認するために linter を使ってください。

### Missing Required Fields

```
Error: Missing required field: project.name
```

answers file に必要なフィールドがすべて含まれていることを確認してください。

### Template Not Found

```
Error: Template 'unknown' not found
```

利用可能な templates を確認するには `gtc wizard --list-templates` を使ってください。
