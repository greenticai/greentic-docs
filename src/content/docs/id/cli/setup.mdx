## Gambaran Umum

Perintah `gtc setup` mengonfigurasikan provider di bundle Greentic Anda. Perintah ini menjalankan flow setup untuk setiap provider yang diaktifkan, mengumpulkan kredensial, dan mengonfigurasi webhook.

## Penggunaan

```bash
gtc setup [OPTIONS] <BUNDLE_PATH>
```

## Opsi

| Opsi | Deskripsi |
|--------|-------------|
| `--answers <FILE>` | Path ke file answers (non-interaktif) |
| `--dry-run` | Pratinjau setup tanpa membuat perubahan |
| `--emit-answers <FILE>` | Buat template answers |
| `--advanced` | Sertakan field opsional dalam template answers |
| `--provider <NAME>` | Setup hanya provider tertentu |
| `--skip <NAME>` | Lewati provider tertentu |
| `-v, --verbose` | Aktifkan output verbose |

## Setup Interaktif

Jalankan setup secara interaktif agar Anda dipandu untuk setiap provider:

```bash
gtc setup ./my-bundle
```

CLI akan meminta:
- Kredensial provider (API key, token)
- URL publik untuk webhook
- Opsi konfigurasi opsional

## Setup Non-Interaktif

### Flow Katalog Demo

Setiap demo dalam katalog [`greentic-demo`](https://github.com/greenticai/greentic-demo) menerbitkan dokumen `*-setup-answers.json` bersama pack-nya. Terapkan dokumen tersebut ke bundle yang Anda buat dengan `gtc wizard`:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

Pola yang sama berlaku untuk demo mana pun, cukup ganti ID demo. Lihat [Menjalankan Demo](/id/getting-started/running-demos/) untuk katalog lengkap beserta deskripsi dan kredensial yang diperlukan.

Anda juga dapat langsung mengarahkan `gtc setup` ke artefak `.gtbundle` yang telah dipublikasikan:

```bash
gtc setup https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo.gtbundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/cloud-deploy-demo-setup-answers.json
```

<Aside type="note">
`gtc setup` mendeteksi tenant bundle secara otomatis dari direktori `tenants/`, sehingga secret ditulis ke URI tenant yang benar bahkan ketika bundle tidak dibuat dengan tenant default `demo`. Perintah ini juga menjalankan **validasi capability dependensi** di seluruh pack dalam bundle dan memperingatkan jika ada capability yang dibutuhkan (misalnya `greentic:state/state-store`) yang masih belum terpenuhi.
</Aside>

### Menghasilkan Template Answers

Pertama, hasilkan template dengan semua field yang diperlukan:

```bash
# Basic template (required fields only)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Full template (including optional fields)
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle
```

## Konfigurasi Khusus Provider

### Telegram

```json
{
  "messaging-telegram": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "123456789:ABCdefGHIjklMNOpqrsTUVwxyz"
  }
}
```

**Cara mendapatkan kredensial:**
1. Kirim pesan ke [@BotFather](https://t.me/botfather) di Telegram
2. Kirim `/newbot` dan ikuti petunjuknya
3. Salin token bot

### Slack

```json
{
  "messaging-slack": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "bot_token": "xoxb-xxx-xxx-xxx",
    "slack_app_id": "A07XXXXXX",
    "slack_configuration_token": "xoxe.xoxp-xxx"
  }
}
```

**Cara mendapatkan kredensial:**
1. Buka [api.slack.com/apps](https://api.slack.com/apps)
2. Buat aplikasi baru atau pilih yang sudah ada
3. Ambil Bot Token dari OAuth & Permissions
4. Ambil App ID dari Basic Information
5. Ambil Configuration Token dari App Manifest

### Microsoft Teams

```json
{
  "messaging-teams": {
    "enabled": true,
    "public_base_url": "https://your-domain.com",
    "app_id": "your-app-id",
    "app_password": "your-app-password",
    "tenant_id": "your-tenant-id"
  }
}
```

### WhatsApp (via Meta)

## Konfigurasi URL Publik

<Aside type="caution">
Provider berbasis webhook memerlukan URL yang dapat diakses publik. `public_base_url` akan diganti secara otomatis saat menggunakan ngrok atau cloudflared dengan `gtc start`.
</Aside>

### Menggunakan ngrok

```bash
# Start ngrok in separate terminal
ngrok http 8080

# Copy the HTTPS URL (e.g., https://abc123.ngrok-free.app)
# Use this as public_base_url in answers
```

### Menggunakan Cloudflared

```bash
# Cloudflared generates URL automatically with gtc start
gtc start ./my-bundle --cloudflared on
```

## Capability Bundle

File `bundle.yaml` dapat mendeklarasikan capability yang membuka perilaku tambahan selama setup. Capability dicantumkan sebagai array identifier capability.

### Bundle Assets (`greentic.cap.bundle_assets.read.v1`)

Ketika capability `greentic.cap.bundle_assets.read.v1` ada di `bundle.yaml`, `gtc setup` membuat direktori `assets/` di root bundle dan melakukan scaffold file aset dari provider pack ke dalamnya.

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

Secara khusus, aset dari provider pack (seperti entri `assets/webchat-gui/`) disalin ke direktori `assets/` level bundle. Ini memberi Anda tempat untuk menyesuaikan aset provider, misalnya skin webchat-gui dan file konfigurasi, tanpa memodifikasi isi pack asli.

**Cara kerjanya:**

- Selama setup, provider pack yang menyertakan direktori aset (misalnya `assets/webchat-gui/`) akan melakukan scaffold file-file tersebut ke direktori `assets/` di root bundle.
- Setelah itu Anda dapat mengedit file-file ini dengan bebas. File aset level bundle akan menimpa default pack saat runtime.
- Ini berguna untuk menyesuaikan tampilan dan nuansa widget webchat, mengubah konfigurasi, atau menerapkan skin bermerek.

<Aside type="tip">
Setelah setup melakukan scaffold aset, Anda dapat memodifikasinya dengan aman. Kustomisasi level bundle Anda akan diprioritaskan dibanding default pack saat runtime memuat aset.
</Aside>

## Eksekusi Flow Setup

Saat Anda menjalankan `gtc setup`, perintah ini:

<Steps>

1. **Memuat konfigurasi bundle**

   Membaca `greentic.demo.yaml` untuk menemukan provider.

2. **Memvalidasi answers**

   Memeriksa bahwa semua field yang diperlukan tersedia.

3. **Menjalankan flow setup**

   Menjalankan `setup_flow` milik setiap provider (misalnya registrasi webhook).

4. **Menyimpan kredensial**

   Menyimpan kredensial ke secrets store yang dikonfigurasi.

5. **Menjalankan verifikasi**

   Menjalankan `verify_flow` untuk mengonfirmasi bahwa setup berhasil.

</Steps>

## Setup Selektif

### Setup Satu Provider

```bash
gtc setup --provider messaging-telegram ./my-bundle
```

### Lewati Provider Tertentu

```bash
gtc setup --skip messaging-teams ./my-bundle
```

### Jalankan Ulang Setup

```bash
# Force re-setup (useful after URL changes)
gtc start ./my-bundle --force-setup
```

## Pemecahan Masalah

### Registrasi Webhook Gagal

```
Error: Failed to register webhook: 401 Unauthorized
```

Periksa bahwa token bot Anda benar dan belum kedaluwarsa.

### Field Wajib Tidak Ada

```
Error: Missing required field 'bot_token' for messaging-telegram
```

Pastikan semua field yang diperlukan ada di file answers Anda.

### URL Publik Tidak Dapat Diakses

```
Error: Webhook verification failed: Connection refused
```

Pastikan URL publik Anda dapat diakses dari internet. Periksa apakah ngrok/cloudflared sedang berjalan.

### Kredensial Tidak Ditemukan

```
Error: Secret 'slack_bot_token' not found
```

Jalankan `gtc setup` untuk mengonfigurasi provider, atau periksa file answers Anda.

## Praktik Terbaik

1. **Gunakan file answers di CI/CD** - Hindari prompt interaktif dalam pipeline otomatis
2. **Jaga keamanan kredensial** - Jangan commit file answers yang berisi token nyata ke git
3. **Gunakan environment variable** - Referensikan secret melalui `$ENV_VAR` di answers
4. **Uji dengan dry-run** - Pratinjau setup sebelum membuat perubahan
5. **Regenerasi saat URL berubah** - Jalankan ulang setup saat URL publik Anda berubah
