## Übersicht

**Greentic Setup** ist die Konfigurationsengine für Greentic-Bundles. Das Ausführen von `gtc setup <bundle>` öffnet ein interaktives Web-Dashboard im Browser, über das Sie Provider-Anmeldedaten konfigurieren, aktuelle Scopes prüfen und den geführten Setup-Assistenten für jeden Provider ausführen können – alles aus einer sicher lokal gehosteten Single-Page-Anwendung.

Für CI- oder SSH-Umgebungen fällt `gtc setup --no-ui <bundle>` auf den Terminal-Dialoger-Flow zurück.

## Starten

```bash
# Web-Dashboard starten (Standard)
gtc setup ./my-bundle

# Headless-Terminal-Modus (CI / SSH)
gtc setup --no-ui ./my-bundle

# Vorgefertigte Antwortdatei ohne UI anwenden
gtc setup --answers answers.json ./my-bundle
```

Beim Ausführen von `gtc setup ./my-bundle` führt das Binary folgende Schritte aus:

1. Lädt und validiert Ihre Bundle-Konfiguration.
2. Startet einen lokalen [Axum](https://docs.rs/axum) HTTP-Server, der auf `127.0.0.1` an einem zufälligen verfügbaren Port gebunden ist.
3. Generiert ein einmaliges Bearer-Token, das nur im Arbeitsspeicher gespeichert wird – wird nie auf die Festplatte geschrieben.
4. Öffnet Ihren Standardbrowser unter `http://127.0.0.1:<port>/?token=<token>`.
5. Stellt die Greentic Setup Single-Page-Anwendung bereit.

Der Server wird automatisch beendet, wenn Sie den Browser-Tab schließen oder die Schaltfläche **Close** im Dashboard verwenden.

## Dashboard

Das Greentic Setup-Dashboard ist eine Single-Page-Anwendung, die mit [Alpine.js](https://alpinejs.dev/) erstellt wurde. Es bietet eine saubere, minimalistische Oberfläche zur Verwaltung Ihrer Bundle-Konfiguration, ohne den Browser zu verlassen.

### Layout

Das Dashboard hat drei Hauptbereiche:

- **Topbar** — zeigt den Bundle-Namen und eine Schaltfläche zum Schließen an, die den Server ordnungsgemäß herunterfährt.
- **Seitenleiste** — enthält drei Scope-Auswahlfelder (Tenant, Umgebung, Team) und das Navigationsmenü.
- **Hauptinhaltsbereich** — rendert die aktive Ansicht.

### Seitenleistennavigation

Drei Dropdown-Auswahlfelder oben in der Seitenleiste ermöglichen es Ihnen, den aktiven Scope-Kontext zu ändern:

| Auswahlfeld | Was es steuert |
|-------------|----------------|
| Tenant | Den Tenant-Namespace für Secrets und Provider-Konfiguration |
| Umgebung | `demo`, `dev`, `staging` oder ein benutzerdefinierter Wert |
| Team | Sub-Team-Scope innerhalb des Tenants |

Der **Manage**-Navigationsbereich zeigt:

| Element | Status |
|---------|--------|
| Overview | Aktiv in Phase 1a |
| Providers | Kommt in Phase 1b |
| Secrets | Kommt in Phase 1b |
| Capabilities | Kommt in Phase 1b |

Mit "1b" gekennzeichnete Elemente sind schreibgeschützte Platzhalter in der aktuellen Version.

### Overview-Ansicht

Overview ist die Standard-Einstiegsansicht. Sie enthält:

- **Willkommensüberschrift** mit dem Bundle-Anzeigenamen.
- **Statistikzeile** mit vier Zusammenfassungs-Chips: Konfigurierte Scopes, Gefundene Provider, Gespeicherte Secrets und Warnungen.
- **Scope-Liste** — eine Karte pro konfiguriertem Scope (Tenant/Umgebung/Team-Kombination), jede zeigt die Provider-Chips, die in diesem Scope eingerichtet wurden.
- **Leerzustand** — wenn noch keine Scopes konfiguriert wurden, erscheint ein Leerzustand-Panel, das Sie auffordert, den ersten Tenant über die Schaltfläche **Add tenant** hinzuzufügen.

## Assistent

Ein Klick auf **Add tenant** (oder eine scope-spezifische **Configure**-Schaltfläche) startet den inline Setup-Assistenten. Der Assistent läuft innerhalb der Dashboard-Shell – er navigiert nicht zu einer separaten Seite.

### Assistent-Flow

Der Assistent sammelt die Konfiguration in einer Reihe von Schritten:

<Steps>

1. **Scope auswählen** — Tenant, Umgebung und Team auswählen oder bestätigen.

2. **Provider wählen** — umschalten, welche Provider für diesen Scope aktiviert werden sollen.

3. **Jeden Provider konfigurieren** — für jeden aktivierten Provider die erforderlichen Felder ausfüllen (API-Schlüssel, öffentliche Webhook-URL usw.).

4. **Überprüfen** — eine Zusammenfassung aller Werte anzeigen, bevor sie gespeichert werden.

5. **Fertigstellen** — Anmeldedaten werden in den Secret-Speicher geschrieben; Overview wird aktualisiert, um den neuen Scope anzuzeigen.

</Steps>

### Feldtypen

Der Assistent unterstützt die folgenden Eingabetypen:

| Typ | Verwendung |
|-----|-----------|
| `text` | Einfache Zeichenkettenwerte (URLs, IDs, Benutzernamen) |
| `password` | Sensible Werte — als maskierte Eingaben gerendert, nach Abschluss aus dem Speicher gelöscht |
| `textarea` | Mehrzeilige Werte (Zertifikat-PEM-Blöcke, JSON-Payloads) |
| `select` | Aufgezählte Optionen |
| `switch` | Boolesche Umschalter (Provider aktivieren/deaktivieren) |

### Navigation

Jeder Schritt hat **Back**-, **Continue**- und **Cancel**-Aktionen. Der letzte Schritt ersetzt **Continue** durch **Finish**. Die Validierung wird pro Schritt vor dem Weitergehen ausgeführt; Fehler werden inline unter dem betreffenden Feld angezeigt.

## Headless-Modus

Wenn Sie über SSH arbeiten, sich in einer CI-Pipeline befinden oder den Browser-Start deaktiviert haben, übergeben Sie `--no-ui`:

```bash
gtc setup --no-ui ./my-bundle
```

Dies ruft den alten Terminal-Dialoger-Flow auf. Alle gleichen Konfigurationswerte werden interaktiv an der Eingabeaufforderung gesammelt, und Secrets werden im gleichen Speicher wie der Dashboard-Assistent gespeichert.

### Antwortdateien

Antwortdateien ermöglichen es Ihnen, das Setup ohne interaktive Eingabeaufforderungen zu automatisieren:

```bash
# Vorlage generieren (nur Pflichtfelder)
gtc setup --dry-run --emit-answers answers.json ./my-bundle

# Vorlage mit optionalen Feldern generieren
gtc setup --dry-run --emit-answers answers.json --advanced ./my-bundle

# Antworten anwenden
gtc setup --answers answers.json ./my-bundle
```

#### Demo-Katalog-Antworten

Jede Demo im [`greentic-demo`](https://github.com/greenticai/greentic-demo)-Katalog veröffentlicht ein `*-setup-answers.json`-Dokument. Wenden Sie es direkt an:

```bash
gtc setup ./quickstart-demo-bundle \
  --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
```

Den vollständigen Katalog mit Beschreibungen und erforderlichen Anmeldedaten finden Sie unter [Demos ausführen](/de/getting-started/running-demos/).

<Aside type="note">
Greentic Setup erkennt den Tenant des Bundles automatisch aus dem Verzeichnis `tenants/`, sodass Secrets selbst dann in den richtigen Tenant-URI geschrieben werden, wenn das Bundle nicht mit dem Standard-Tenant `demo` erstellt wurde. Es führt auch **Abhängigkeits-Capabilities-Validierung** über jeden Pack im Bundle durch und warnt, wenn eine erforderliche Capability (z. B. `greentic:state/state-store`) noch nicht erfüllt ist.
</Aside>

## Sicherheit

Greentic Setup verfolgt einen Security-First-Ansatz für alle lokalen UI-Sitzungen:

| Kontrolle | Implementierung |
|-----------|----------------|
| Nur Loopback | Server bindet ausschließlich an `127.0.0.1` — keine Netzwerkexposition |
| Zufälliger Port | Beim Start wird ein zufälliger verfügbarer Port gewählt — kein fester Port zum Erraten |
| Bearer-Token | Einmaliges Token, im Arbeitsspeicher generiert, beim ersten Laden über URL-Query-Parameter übergeben, dann nur in der Browser-Sitzung gespeichert |
| Origin-Durchsetzung | Jede API-Anfrage wird gegen eine `Origin`-Allowlist validiert; Cross-Origin-Anfragen werden abgelehnt |
| Keine Secrets in Logs | Anmeldedaten werden nie in stdout, stderr oder Log-Dateien geschrieben |
| Keine Secrets auf der Festplatte | Bearer-Token und Assistenten-Antworten werden nur im Arbeitsspeicher gehalten; `Zeroize` wird nach der Verwendung auf sensiblen Strukturen aufgerufen |
| Same-Origin-Assets | Alle JS-, CSS- und Font-Assets werden vom gleichen lokalen Server bereitgestellt — keine externen CDN-Aufrufe |
| Kein Telemetrie | Das Dashboard führt keine ausgehenden Netzwerkaufrufe durch |

## Bundle-Capabilities

Die Datei `bundle.yaml` kann Capabilities deklarieren, die während des Setups zusätzliches Verhalten freischalten.

### Bundle-Assets (`greentic.cap.bundle_assets.read.v1`)

Wenn diese Capability deklariert ist, erstellt Greentic Setup ein `assets/`-Verzeichnis im Bundle-Root und kopiert Provider-Pack-Assets hinein (z. B. `assets/webchat-gui/`). Sie können diese Dateien dann frei ändern; Bundle-level Asset-Dateien überschreiben Pack-Standards zur Laufzeit.

```yaml title="bundle.yaml"
capabilities:
  - greentic.cap.bundle_assets.read.v1
```

<Aside type="tip">
Nachdem das Setup die Assets erstellt hat, haben Ihre Bundle-level-Anpassungen beim Laden durch die Laufzeit Vorrang vor den Pack-Standards.
</Aside>

## Roadmap

Greentic Setup Phase 1a liefert die Dashboard-Shell, die Overview-Ansicht und den Inline-Assistenten. Kommende Arbeiten in **Phase 1b** umfassen:

- **Providers** — alle Packs im Bundle durchsuchen, einzelne Provider aktivieren oder deaktivieren, das Setup für einen einzelnen Provider erneut ausführen, ohne andere zu berühren.
- **Secrets** — vollständiges CRUD für den Secret-Speicher: einzelne Secret-Einträge pro Scope erstellen, lesen (maskiert), aktualisieren und löschen.
- **Capabilities** — Capability-Flags pro Pack umschalten und den Abhängigkeitsgraph inspizieren.
- **Rebuild-Trigger** — ein Hot-Reload des laufenden Bundles vom Dashboard aus anfordern, ohne die vollständige Laufzeit neu zu starten.

## Fehlerbehebung

### Browser öffnet sich nicht

Wenn Ihr System den automatischen Browser-Start nicht unterstützt, kopieren Sie die in das Terminal ausgegebene URL und fügen Sie sie manuell in Ihren Browser ein:

```
Greentic Setup running at http://127.0.0.1:52341/?token=...
```

### Webhook-Registrierung fehlgeschlagen

```
Error: Failed to register webhook: 401 Unauthorized
```

Überprüfen Sie, ob Ihr Bot-Token korrekt ist und nicht abgelaufen ist.

### Pflichtfeld fehlt

```
Error: Missing required field 'bot_token' for messaging-telegram
```

Stellen Sie sicher, dass alle Pflichtfelder im Assistenten ausgefüllt oder in Ihrer Antwortdatei vorhanden sind.

### Öffentliche URL nicht erreichbar

```
Error: Webhook verification failed: Connection refused
```

Ihre öffentliche URL muss aus dem Internet erreichbar sein. Überprüfen Sie, ob ngrok oder cloudflared läuft.

### Anmeldedaten nicht gefunden

```
Error: Secret 'slack_bot_token' not found
```

Führen Sie `gtc setup` aus, um den Provider zu konfigurieren, oder überprüfen Sie Ihre Antwortdatei.
