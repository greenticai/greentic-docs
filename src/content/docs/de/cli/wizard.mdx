## Überblick

Der Befehl `gtc wizard` erstellt neue Greentic-Bundles aus Wizard-Antworten. Er scaffoldet die vollständige Bundle-Struktur einschließlich Konfigurationsdateien, Provider-Setups und App-Vorlagen.

## Verwendung

```bash
gtc wizard [OPTIONS]
```

## Optionen

| Option | Beschreibung |
|--------|-------------|
| `--answers <FILE \| URL>` | Pfad oder URL zur Wizard-Antwortdatei (JSON oder YAML). Unterstützt lokale Pfade und entfernte `http://`- / `https://`-URLs. |
| `--dry-run` | Erzeugte Dateien in der Vorschau anzeigen, ohne sie zu schreiben |
| `--output <DIR>` | Ausgabeverzeichnis (Standard: aktuelles Verzeichnis) |
| `--template <NAME>` | Eine bestimmte Vorlage verwenden |
| `-v, --verbose` | Ausführliche Ausgabe aktivieren |

## Wizard-Antwortdatei

### JSON-Format

```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0",
    "description": "Customer support digital worker"
  },
  "providers": {
    "messaging": ["telegram", "slack"],
    "events": ["webhook", "timer"]
  },
  "apps": [
    {
      "name": "support-bot",
      "template": "customer-service"
    }
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [
        {
          "id": "default",
          "channels": ["telegram", "slack"]
        }
      ]
    }
  ]
}
```

### YAML-Format

```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"
  description: Customer support digital worker

providers:
  messaging:
    - telegram
    - slack
  events:
    - webhook
    - timer

apps:
  - name: support-bot
    template: customer-service

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - telegram
          - slack
```

### Capabilities

Das Array `capabilities` in der Wizard-Antwortdatei steuert Bundle-weite Funktionen, die beim Erstellen des Bundles aktiviert werden. Capabilities bestimmen, welche zusätzlichen Scaffold-Schritte der Wizard über die Standard-Bundle-Struktur hinaus ausführt.

```json title="create-answers.json"
{
  "capabilities": ["greentic.cap.bundle_assets.read.v1"]
}
```

**Verfügbare Capabilities:**

| Capability | Beschreibung |
|------------|-------------|
| `greentic.cap.bundle_assets.read.v1` | Aktiviert Bundle-weite Asset-Überschreibungen. Wenn vorhanden, scaffoldet der Wizard automatisch `webchat-gui`-Skin-Assets aus Provider-Packs in das Verzeichnis `assets/` des Bundles. |

Wenn `greentic.cap.bundle_assets.read.v1` enthalten ist, enthält das erzeugte Bundle ein Verzeichnis `assets/` mit Dateien, die aus Provider-Packs kopiert wurden, etwa `webchat-gui`-Skins und Konfigurationen. Diese Dateien auf Bundle-Ebene können angepasst werden und überschreiben zur Laufzeit die Standardwerte des Packs.

<Aside type="tip">
Das Array `capabilities` wird in `bundle.yaml` geschrieben und später auch von `gtc setup` berücksichtigt, wenn das Bundle konfiguriert wird. Details zum Laufzeitverhalten finden Sie im [Capabilities-Abschnitt von gtc setup](/de/cli/setup/#bundle-capabilities).
</Aside>

## Beispiele

### Demo-Katalog-Bundles

Das Repository [`greentic-demo`](https://github.com/greenticai/greentic-demo) veröffentlicht für jede Demo in seinen Release-Artefakten ein `*-create-answers.json` und ein `*-setup-answers.json`. Verwenden Sie diese direkt, um ein lokales Bundle zu erstellen, das Sie anschließend weiter bearbeiten können:

```bash
gtc wizard --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-create-answers.json
```

Fahren Sie dann mit Setup und Start fort:

```bash
gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
gtc start ./quickstart-demo-bundle
```

Ersetzen Sie die Demo-ID durch eine beliebige der veröffentlichten Demos (`quickstart`, `quickstart-event`, `hr-onboarding`, `helpdesk-itsm`, `sales-crm`, `supply-chain`, `incident`, `redbutton`, `cloud-deploy-demo`, `weather-mcp-demo`, ...). Der vollständige Katalog mit Beschreibungen und erforderlichen Zugangsdaten befindet sich auf der Seite [Demos ausführen](/de/getting-started/running-demos/).

<Aside type="tip">
Der Wizard führt eine **Capability-basierte Abhängigkeitsauflösung** durch. Wenn das Create-Answers-Dokument der Demo auf einen Provider verweist, der `required_capabilities` deklariert (z. B. Messaging-Provider, die `greentic:state/state-store` benötigen), fügt der Wizard automatisch ein passendes Pack wie `state-memory` zum erzeugten Bundle hinzu. Sie müssen es nicht manuell auflisten.
</Aside>

### Grundlegende Verwendung

```bash
# Interaktiver Modus
gtc wizard

# Mit Answers-Datei
gtc wizard --answers wizard-answers.yaml
```

### Vorschaumodus

```bash
# Anzeigen, was erzeugt würde
gtc wizard --answers wizard-answers.yaml --dry-run
```

### Entfernte Answers-Datei

Das Flag `--answers` akzeptiert `http://`- und `https://`-URLs. Dadurch können gemeinsam genutzte Answer-Dokumente zentral gespeichert und aus CI/CD-Pipelines referenziert werden.

```bash
# Answers von einer entfernten URL abrufen
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml

# Mit dry-run kombinieren, um vor dem Erzeugen eine Vorschau anzuzeigen
gtc wizard --answers https://config.example.com/teams/support/wizard-answers.yaml --dry-run
```

<Aside type="tip">
Die Unterstützung entfernter URLs ist in CI/CD-Pipelines nützlich, wenn Wizard-Antworten in einem gemeinsamen Repository oder Artefaktserver gespeichert und zur Build-Zeit abgerufen werden.
</Aside>

### Benutzerdefiniertes Ausgabeverzeichnis

```bash
gtc wizard --answers wizard-answers.yaml --output ./my-project
```

### Vorlagen verwenden

```bash
# Verfügbare Vorlagen auflisten
gtc wizard --list-templates

# Bestimmte Vorlage verwenden
gtc wizard --template customer-service --answers answers.yaml
```

## Bundle-Capabilities

Der Wizard fragt nach optionalen Bundle-Capabilities, die zusätzliche Funktionen aktivieren. Jede Capability schaltet einen bestimmten Satz an Assets und Verhaltensweisen im erzeugten Bundle frei.

### Interaktive Eingaben

Wenn Sie `gtc wizard` interaktiv ausführen, sehen Sie:

```
? Enable bundle-level assets (./assets/)? [Y/n]
? Enable OAuth login for WebChat GUI? [y/N]
? Enable multi-language for WebChat GUI? [y/N]
? Enable embeddable WebChat widget? [y/N]
```

### Capability-Referenz

| Prompt | Capability ID | Was sie bewirkt |
|--------|---------------|-------------|
| Enable bundle-level assets | `greentic.cap.bundle_assets.read.v1` | Erstellt das Verzeichnis `./assets/` und scaffoldet Standard-Skin-Dateien aus Provider-Packs in Tenant-spezifische Verzeichnisse |
| Enable OAuth login for WebChat GUI | `greentic.cap.webchat.oauth.v1` | Fügt OAuth/OIDC-Authentifizierungsprovider-Konfiguration zu Tenant-Konfigurationsdateien hinzu |
| Enable multi-language for WebChat GUI | `greentic.cap.webchat.i18n.v1` | Aktiviert den Locale-Picker in WebChat und scaffoldet Locale-JSON-Dateien unter `./assets/` |
| Enable embeddable WebChat widget | `greentic.cap.webchat.embed.v1` | Erzeugt Embed-HTML-Snippets (Fullpage-Link, Inline-iframe, Chat Bubble) pro Tenant |

### Capabilities in der Answers-Datei setzen

Anstatt Prompts interaktiv zu beantworten, können Sie Capabilities direkt in der Answers-Datei festlegen, indem Sie ein Array `capabilities` hinzufügen:

<Tabs>
  <TabItem label="JSON">
```json title="wizard-answers.json"
{
  "project": {
    "name": "my-digital-worker",
    "version": "1.0.0"
  },
  "providers": {
    "messaging": ["webchat", "telegram"]
  },
  "capabilities": [
    "greentic.cap.bundle_assets.read.v1",
    "greentic.cap.webchat.oauth.v1",
    "greentic.cap.webchat.i18n.v1",
    "greentic.cap.webchat.embed.v1"
  ],
  "tenants": [
    {
      "id": "demo",
      "name": "Demo Tenant",
      "teams": [{ "id": "default", "channels": ["webchat"] }]
    }
  ]
}
```
  </TabItem>
  <TabItem label="YAML">
```yaml title="wizard-answers.yaml"
project:
  name: my-digital-worker
  version: "1.0.0"

providers:
  messaging:
    - webchat
    - telegram

capabilities:
  - greentic.cap.bundle_assets.read.v1
  - greentic.cap.webchat.oauth.v1
  - greentic.cap.webchat.i18n.v1
  - greentic.cap.webchat.embed.v1

tenants:
  - id: demo
    name: Demo Tenant
    teams:
      - id: default
        channels:
          - webchat
```
  </TabItem>
</Tabs>

<Aside type="tip">
Capabilities sind additiv. Das Aktivieren von `greentic.cap.webchat.embed.v1` erfordert implizit `greentic.cap.bundle_assets.read.v1`, weil die Embed-Snippets in das Verzeichnis `./assets/` geschrieben werden. Der Wizard behandelt diese Abhängigkeit automatisch.
</Aside>

Ausführliche Informationen zu Capabilities, Scaffold-Verhalten und dem Overlay-Mechanismus finden Sie unter [Bundle-Assets](/de/greentic-docs/concepts/bundle-assets/).

## Workflow

<Steps>

1. **Answers-Datei erstellen**

   Definieren Sie Ihre Projektkonfiguration in JSON oder YAML.

2. **Wizard ausführen**

   ```bash
   gtc wizard --answers wizard-answers.yaml
   ```

3. **Erzeugte Dateien prüfen**

   Kontrollieren Sie die erzeugte Bundle-Struktur.

4. **Provider konfigurieren**

   ```bash
   gtc setup ./my-digital-worker
   ```

5. **Runtime starten**

   ```bash
   gtc start ./my-digital-worker
   ```

</Steps>

## Erzeugte Flows anpassen

Der Wizard erzeugt grundlegende Flow-Vorlagen. Passen Sie sie nach der Generierung an:

```yaml title="apps/support-bot/flows/on_message.ygtc"
name: on_message
version: "1.0"
description: Handle incoming messages

nodes:
  # Add your custom nodes here
  - id: analyze
    type: llm
    config:
      model: "gpt-4"
      prompt: "Analyze: {{message}}"
    next: respond

  - id: respond
    type: reply
    config:
      message: "{{analysis_result}}"

triggers:
  - type: message
    target: analyze
```

<Aside type="tip">
Verwenden Sie zuerst `--dry-run`, um die erzeugte Struktur in der Vorschau zu prüfen, bevor Sie sie auf die Festplatte schreiben.
</Aside>

## Component scaffolding: `http_client` capability

When scaffolding a new WASM component (for example via `greentic-component new`), the wizard can mark the component as needing outbound HTTP by setting `http_client: true`. This writes the capability declaration into the component manifest.

<Aside type="caution">
**Desktop runner HTTP gate.** Declaring `http_client: true` is a capability _request_ — the runtime must grant it before the component can actually open outbound sockets. On the **desktop runner profile**, honouring this manifest flag has been a no-op until the HTTP gate fix lands: components that declared `http_client: true` had no `wasi:http/outgoing-handler` implementation linked, and runtime HTTP calls failed.

As of Greentic runner `N.N.N`, the desktop profile honours the manifest and grants `wasi:http/outgoing-handler@0.2.0` automatically when `http_client: true` is declared. {/* TODO: confirm runner version once Bug 2 PR merges */}

**Until the fix is released, the workaround is** to ship your component with a custom WIT world that imports `wasi:http/outgoing-handler@0.2.0` directly, rather than relying on the scaffolded manifest flag. Once Bug 2 is merged and you are on a runner that includes it, you can drop the custom world and use `http_client: true` as intended.

For components that emit Adaptive Cards via HTTP fetches (e.g. RAG lookups before returning an `attachments[]` array), see the [WebChat provider's Attachment envelope shape](/providers/messaging/webchat/#attachment-envelope-shape) section for the expected component-output layout.
</Aside>

## Fehlerbehebung

### Ungültige Answers-Datei

```
Error: Failed to parse answers file
```

Stellen Sie sicher, dass Ihr JSON/YAML gültig ist. Verwenden Sie einen Linter, um die Syntax zu prüfen.

### Erforderliche Felder fehlen

```
Error: Missing required field: project.name
```

Prüfen Sie, dass alle erforderlichen Felder in Ihrer Answers-Datei vorhanden sind.

### Vorlage nicht gefunden

```
Error: Template 'unknown' not found
```

Verwenden Sie `gtc wizard --list-templates`, um die verfügbaren Vorlagen anzuzeigen.
