This guide gets you from zero to a running Greentic digital worker in a few minutes. The fastest path is to launch one of the published demos from the [`greentic-demo`](https://github.com/greenticai/greentic-demo) catalog тАФ every demo uses the same three-step flow.

<Aside type="tip">
Want to browse all available demos and see exactly what each one does? Jump to [Running Demos](/getting-started/running-demos/).
</Aside>

## Prerequisites

Before you begin, ensure you have:

- **`gtc` CLI** installed тАФ see [Installation](/getting-started/installation/)
- **Rust** 1.91 or later (only needed if you build from source)
- A **public URL** for webhook providers (e.g. `ngrok` or `cloudflared`) тАФ not required for the WebChat-only quickstart

## Launch the quickstart demo

The `quickstart` demo is the smallest end-to-end bundle: a welcome card, an about card, and an echo interaction over WebChat. It needs no external accounts.

<Steps>

1. **Create the bundle** from the published create-answers document:

   ```bash
   gtc wizard --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-create-answers.json
   ```

   The wizard downloads the demo pack, resolves capability dependencies (e.g. it pulls in `state-memory` automatically because the WebChat provider needs `greentic:state/state-store`), and writes a local bundle directory.

2. **Configure providers** with the published setup-answers document:

   ```bash
   gtc setup ./quickstart-demo-bundle --answers https://github.com/greenticai/greentic-demo/releases/latest/download/quickstart-setup-answers.json
   ```

   Setup auto-detects the bundle's tenant, persists secrets per provider, and validates that every required capability is satisfied.

3. **Start the runtime**:

   ```bash
   gtc start ./quickstart-demo-bundle
   ```

   The runner prints the public WebChat URL тАФ open it in a browser and you should see the welcome card.

</Steps>

<Aside type="note">
The same three-step flow works for every demo in the catalog. To try a different one, just substitute the demo ID тАФ for example `helpdesk-itsm`, `hr-onboarding`, or `weather-mcp-demo`. See [Running Demos](/getting-started/running-demos/) for the complete list.
</Aside>

## Building your own bundle

When you are ready to move beyond the published catalog, the same three commands apply тАФ they just consume your own files instead of release URLs:

<Steps>

1. **Create a bundle** with the wizard. You can run it interactively or with a local answers document:

   ```bash
   gtc wizard
   # or
   gtc wizard --answers ./my-create-answers.json
   ```

2. **Configure providers**, again either interactively or non-interactively:

   ```bash
   gtc setup ./my-bundle
   # or
   gtc setup ./my-bundle --answers ./my-setup-answers.json
   ```

3. **Start the runtime**:

   ```bash
   gtc start ./my-bundle
   ```

</Steps>

See [gtc wizard](/cli/wizard/) and [gtc setup](/cli/setup/) for the full reference, including capability flags and answers-document schemas.

## Example: Hello World Flow

Create a simple flow that responds to messages:

```yaml title="flows/hello.ygtc"
name: hello_world
version: "1.0"
description: A simple greeting flow

nodes:
  - id: greet
    type: reply
    config:
      message: "Hello! I'm your digital worker."

triggers:
  - type: message
    pattern: "hello"
    target: greet
```

## Configuration File

The main configuration file (`greentic.demo.yaml`) defines your setup:

```yaml title="greentic.demo.yaml"
name: my-digital-worker
version: "1.0"

providers:
  messaging-telegram:
    pack: "providers/messaging/messaging-telegram.gtpack"
    setup_flow: "setup_default"

apps:
  my-app:
    path: "apps/my-app"
    default_flow: "on_message"

tenants:
  demo:
    name: Demo Tenant
    teams:
      default:
        channels:
          telegram:
            provider: messaging-telegram
```
